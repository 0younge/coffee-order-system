# 코드 리뷰

## 리뷰 관점 (순서대로)

1. **정확성** — 요구사항 충족, 엣지 케이스(빈 값·경계·동시성), 에러 경로.
2. **단순성** — 과설계 여부. 더 적은 코드로 같은 일을 할 수 있으면 지적한다 (하네스 2원칙).
3. **외과성** — 목적과 무관한 변경이 섞였는지 (하네스 3원칙).
4. **테스트** — 변경에 상응하는 테스트가 있는지, 그 테스트가 실제로 실패를 잡는지.
5. **보안** — 입력 검증, 시크릿 하드코딩, 인젝션, 권한.
6. **성능** — 명백한 문제(N+1, 루프 내 IO)만. 추측성 최적화 지적 금지.

## 보고 형식

심각도별로 묶는다. 각 항목은 `파일:줄 — 문제 — 제안` 한 줄:

- **[must]** 머지 전 반드시 수정 (버그·보안)
- **[should]** 수정 권장 (설계·테스트 누락)
- **[nit]** 사소/취향 (선택)

잘한 점도 1~2개 언급한다.

## 원칙

- 리뷰는 보고까지. 코드는 사용자 지시 후에만 수정한다.
- 확신이 없는 지적은 단정하지 말고 질문 형태로 쓴다.
