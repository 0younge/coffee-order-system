# 테스트 — 범용 완료 기준

여러 언어를 쓰는 프로젝트이므로 특정 프레임워크가 아닌 아래 범용 기준으로 "완료"를 판정한다.

## 완료 체크리스트 (전부 통과해야 완료)

1. 스택을 감지해 해당 명령을 실행한다:

   | 감지 파일 | 테스트 | 린트/검사 |
   |---|---|---|
   | package.json | `npm test` (scripts 확인) | `npm run lint`, `npm run build` |
   | pyproject.toml / requirements.txt | `pytest` | `ruff check` |
   | go.mod | `go test ./...` | `go vet ./...` |
   | Cargo.toml | `cargo test` | `cargo clippy` |
   | 그 외 | README 또는 `.github/workflows/`에서 명령 확인 | |

2. 변경 코드에 대응하는 테스트를 실행한다. 전체가 오래 걸리면 관련 테스트 먼저, 커밋 전에 전체.
3. 프론트엔드 추가 기준: 빌드 성공 + 브라우저 콘솔 에러 0 + 변경 화면의 수동 확인 절차 안내.
4. 결과를 수치 그대로 보고한다 (통과/실패 개수). 실패를 숨기거나 "될 겁니다"로 끝내지 않는다.

## 테스트 작성 — 커버리지 기준 (질적)

코드 작성·변경과 테스트 작성은 한 작업이다. **테스트 없는 코드는 미완료로 판정한다.**

- 새 기능 = 정상 경로 1개 + 실패/경계 경로 1개 이상.
- 변경·추가한 공개 함수/메서드와 조건 분기마다 대응 테스트를 둔다.
  판정 질문: "이 코드를 지우면 실패하는 테스트가 있는가?" — 없으면 테스트를 추가한다.
- 버그 수정 = 재현 테스트 먼저 작성 → 수정 → 통과 확인.
- 테스트가 곤란한 코드(외부 API, UI 렌더링 등)는 생략 사유와 수동 확인 절차를 보고한다.
  조용히 넘어가지 않는다.
- 테스트 러너가 없으면 도입을 제안하고, 승인 전까지는 최소 실행 스크립트로 검증한다.
- 수치 커버리지 목표는 스택 확정 후 측정 도구·명령과 함께 이 스킬에 추가한다.

## 금지

- 통과를 위해 테스트를 약화(assertion 삭제, skip 처리)하지 않는다. 불가피하면 사유를 보고하고 승인받는다.
