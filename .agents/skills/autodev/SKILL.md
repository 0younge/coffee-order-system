# 자동 개발

명시적으로 호출된 경우에만 승인된 문서 범위를 GitHub 이슈 단위로 끝까지 구현한다. 사용자의 실행 확인 전에는 읽기 작업만 하고, 확인 후에는 이 문서에 적힌 범위에서 추가 승인 없이 진행한다.

## 1. 실행 전 계획과 확인

1. 다음 조건을 먼저 확인한다.
   - 현재 브랜치가 `main`이다.
   - `git status --short`가 비어 있다.
   - GitHub 저장소와 인증을 식별할 수 있다.
   - 선제적으로 `git fetch`나 `git pull`을 실행하지 않는다.
2. `docs/requirements-traceability.md`를 작업 대장 정본으로 읽고, `docs/prd.md`, `docs/api-spec.md`, `docs/architecture.md`, `docs/erd.md`, `docs/test-strategy.md`, `docs/adr/`, `README.md`와 실제 코드를 함께 대조한다.
3. 다음 항목은 완료 대상에서 제외한다.
   - 문서에 명시된 제외 범위
   - 예약 상태의 테스트와 기능
   - 합의되지 않은 `TBD` 성능·운영 목표
4. 기존 GitHub 이슈를 읽어 요구사항 ID와 목적이 같은 이슈를 중복 생성하지 않도록 계획한다.
5. 사용자에게 아래 내용을 한 번에 제시하고 실행 여부를 한 번만 확인한다.
   - 저장소·브랜치·현재 구현 기준선
   - 완료 대상과 제외 대상
   - 추적 이슈와 전체 작업 이슈 계획, 의존 순서
   - GitHub 이슈·라벨 쓰기, Docker 기동, 로컬 커밋, `main` push 권한
   - 완료 게이트와 중단 조건
6. 사용자가 승인하지 않으면 어떤 파일이나 외부 상태도 변경하지 않고 종료한다. 승인하면 아래 절차 전체를 수행하라는 명시적 지시로 간주한다.

## 2. GitHub 작업 대장 만들기

1. `autodev`, `in-progress`, `blocked` 라벨을 재사용하고, 없으면 생성한다.
2. 전체 진행을 관리할 추적 이슈 하나와 독립적으로 검증 가능한 작업 이슈 전체를 생성한다.
3. 추적 이슈에 완료 대상, 제외 대상, 작업 이슈 체크리스트, 의존 관계, 최종 통합 검증 이슈를 기록한다.
4. 각 작업 이슈에 다음을 기록한다.
   - 관련 요구사항·정책 ID와 문서 링크
   - 현재 구현 근거와 빠진 항목
   - 수용 기준과 계획된 테스트 ID
   - 선행 이슈와 완료 게이트
5. 의존 관계에 따라 한 번에 한 작업 이슈만 구현한다. 이슈 시작 시 `in-progress` 라벨과 시작 댓글을 남긴다.
6. 구현 중 승인된 문서 범위의 필수 작업을 새로 발견하면 작업 이슈를 추가하고 추적 이슈와 의존 관계를 갱신한다.
7. 문서가 충돌하거나 해석이 여러 개면 blocker 이슈를 만들고 종속 이슈를 막는다. 아직 로컬 커밋이 생기지 않았다면 실행 가능한 독립 이슈를 계속한다.

## 3. 이슈 구현과 커밋

1. 이슈 시작 커밋 SHA를 기록하고 요구사항·관련 코드를 읽어 변경 범위를 요약한 뒤 구현한다.
2. 승인된 문서나 ADR에 이름이 명시된 의존성만 자동으로 추가한다. 그 밖의 의존성, 새 제품 범위, 미승인 ADR이 필요하면 blocker로 처리한다.
3. 버그는 재현 테스트를 먼저 작성하고, 새 기능과 변경에는 정상·실패·경계 테스트를 함께 작성한다.
4. 한 이슈 안에서 목적이 분리되는 변경을 여러 응집된 커밋으로 나눈다.
   - 코드 커밋에는 해당 동작을 검증하는 테스트를 함께 포함한다.
   - 추적성 문서 갱신은 필요하면 별도 커밋으로 분리한다.
   - 커밋 제목은 프로젝트 `commit` 스킬을 따르고 본문에 `관련 이슈: #<번호>`를 기록한다.
   - `--amend`, rebase, force push로 기록을 다시 쓰지 않는다.
5. 이슈가 리뷰·검증을 통과하기 전에는 push하지 않는다.

## 4. 서브 에이전트 리뷰

1. 일반 작업은 읽기 전용 리뷰어 한 명에게 맡긴다.
2. 동시성·보안·외부 HTTP 연동 작업은 서로 독립적인 읽기 전용 리뷰어 두 명에게 맡긴다.
   - 한 명은 요구사항·정확성·트랜잭션·동시성을 검토한다.
   - 다른 한 명은 테스트·보안·단순성·변경 범위를 검토한다.
3. 리뷰어에게 이슈 본문, 관련 원문 문서, 이슈 시작 SHA부터 현재 `HEAD`까지의 diff와 테스트 결과만 전달한다. 결론이나 기대 지적을 알려주지 않는다.
4. 리뷰어는 파일을 수정하지 않고 프로젝트 `review` 스킬의 `[must]`, `[should]`, `[nit]` 형식으로 보고한다.
5. 주 에이전트는 `[must]`와 `[should]`를 모두 수정하고 새 커밋으로 남긴 뒤 관련 테스트와 전체 게이트를 다시 실행한다. 이 스킬의 실행 승인은 리뷰 지적 수정에 대한 사용자 지시로 간주한다.
6. `[must]`와 `[should]`가 모두 0개가 될 때까지 변경된 전체 이슈 diff를 다시 리뷰한다. `[nit]`은 수정하거나 남기는 이유를 이슈에 기록한다.
7. 같은 문제가 세 번 반복되거나 진전이 없으면 해당 이슈를 `blocked`로 처리한다. 로컬 커밋이 이미 생겼다면 다른 이슈로 넘어가지 말고 전체 자동화를 중단한다.

## 5. 이슈 완료 게이트

1. 개발 중에는 변경에 직접 관련된 테스트를 먼저 실행한다.
2. 이슈 완료 전 Docker 상태를 확인한다. MySQL이 준비되지 않았으면 `docker compose up -d --wait`를 실행한다. Docker 엔진을 사용할 수 없거나 health 확인이 실패하면 blocker로 처리한다.
3. 다음 전체 검증을 실행한다.

   ```bash
   ./gradlew clean check build
   ```

4. `./gradlew bootRun`으로 애플리케이션 시작을 확인하고, health endpoint가 구현돼 있으면 응답까지 확인한다. 에이전트가 시작한 프로세스만 종료한다.
5. README와 테스트 전략에 필요한 로컬 검증 환경 변수는 해당 프로세스에만 주입한다. 값을 파일·로그·이슈·커밋에 남기지 않는다.
6. 테스트 전체·성공·실패·제외 수, 실행 명령, 애플리케이션 시작 결과를 그대로 기록한다. 실패나 제외가 있으면 완료로 판정하지 않는다.
7. 구현·테스트·문서·마이그레이션의 이름과 상태가 일치하도록 `docs/requirements-traceability.md`의 상태와 증거를 같은 이슈에서 갱신한다.

## 6. Push와 이슈 종료

1. 리뷰와 완료 게이트 통과 후 전체 diff, 커밋 목록, 시크릿 포함 여부를 다시 확인한다.
2. 원격을 선제적으로 동기화하지 않고 일반 `git push origin main`을 실행한다.
3. `non-fast-forward`면 pull·merge·rebase·force push를 하지 않고 이슈를 `blocked`로 처리한 뒤 전체 자동화를 중단한다.
4. push된 원격 커밋을 확인한다. GitHub Actions가 있으면 해당 커밋의 필수 실행이 성공할 때까지 이슈를 닫지 않는다.
5. push나 CI가 실패하면 원인을 분석하고 수정·재리뷰·재검증·새 커밋·push 순서로 재시도한다. 같은 실패가 세 번 반복되면 `blocked`로 처리하고 전체 자동화를 중단한다.
6. 원격 반영과 CI 성공을 확인한 뒤 검증 결과와 커밋을 종료 댓글에 기록하고 작업 이슈를 닫는다. CI가 없으면 원격 반영 확인 후 닫는다.

## 7. Blocker 기록

1. 막힌 이슈에 `blocked` 라벨을 붙이고 `in-progress`를 제거한다.
2. 원문 오류, 영향받는 요구사항·종속 이슈, 시도 횟수, 재현 방법, 해제 조건을 댓글과 추적 이슈에 기록한다.
3. 로컬 커밋이 없는 계획 단계 blocker라면 종속 이슈를 건너뛰고 실행 가능한 독립 이슈를 계속한다.
4. 로컬 커밋이 생긴 이슈의 blocker, 원격 분기, Docker 엔진 실패처럼 안전하게 우회할 수 없는 blocker가 발생하면 전체 자동화를 중단하고 현재 상태를 보고한다.

## 8. 전체 완료 판정

1. 모든 작업 이슈가 끝나면 최종 통합 검증 이슈를 실행한다.
2. 현재 범위의 요구사항·정책이 모두 `검증됨`인지, 필수 테스트가 구현됐는지, 실패·제외가 0인지 전체 추적성을 점검한다.
3. Docker MySQL에서 전체 테스트·검사·빌드·애플리케이션 시작을 다시 확인한다.
4. 읽기 전용 리뷰어 두 명에게 전체 저장소를 독립적으로 검토하게 하고 `[must]`와 `[should]`를 모두 해소한다.
5. 최종 검증 변경도 동일한 커밋·push·CI 절차로 처리한다.
6. 모든 작업·blocker 이슈가 닫히고 최종 통합 검증이 통과한 경우에만 추적 이슈를 닫고 완료를 보고한다.
