# ADR — 아키텍처 결정 기록

"왜 이렇게 만들었지?"에 답하는 기록. 결정 1개 = 파일 1개.
파일명은 `docs/adr/NNNN-영어-케밥-제목.md` (한글 파일명은 NFD 이슈로 금지, 제목·본문은 한국어).

## 작성 시점 — 하나라도 해당하면 쓴다

- 언어·프레임워크·주요 라이브러리·인프라의 선택 또는 교체
- 데이터 모델, API 계약, 모듈 경계 등 되돌리기 비싼 구조 결정
- 트레이드오프가 있어 나중에 누군가(사람·AI)가 "왜?"라고 물을 선택
- 기존 ADR을 뒤집는 결정

변수명·파일 위치 같은 구현 디테일은 기록하지 않는다.

## 절차

1. `docs/adr/`의 마지막 번호 +1을 쓴다 (0001부터 4자리). 디렉토리가 없으면 만든다.
2. 아래 템플릿으로 작성한다. 상태는 `제안됨`으로 시작.
3. 결정자 전원 합의 후 `승인됨`으로 바꾼다. 1인 프로젝트면 본인 확정 = 승인.
   합의 과정의 이견·소수 의견은 "고려한 대안"에 한 줄로 남긴다.
4. 승인된 ADR 본문은 고치지 않는다. 결정을 뒤집을 때는 새 ADR을 쓰고,
   기존 ADR의 상태를 `대체됨 (→ NNNN)`으로 바꾼다.
5. AI 에이전트: 구조에 영향 주는 작업 전에 관련 ADR을 읽는다.
   ADR과 어긋나는 요청을 받으면 해당 ADR을 근거로 먼저 확인 질문한다.

## 템플릿

```markdown
# NNNN. <결정 제목>

- 상태: 제안됨 | 승인됨 | 대체됨 (→ NNNN)
- 날짜: YYYY-MM-DD
- 결정자: <참여자 이름 — 1인이면 본인만>

## 맥락

<결정이 필요해진 문제·제약. 2~5문장.>

## 고려한 대안

<`대안 — 핵심 장점 / 탈락 이유` 형식으로 대안마다 한 줄. 실질 대안이 없었으면 "없음".>

## 결정

<선택한 것과 핵심 근거. "~한다" 단정형.>

## 결과

<좋아지는 것 / 감수하는 것 / 되돌리려면 필요한 비용.>
```
