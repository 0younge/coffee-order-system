# 하네스 관리

## 규칙 추가·수정 원칙

1. 규칙은 **AGENTS.md에만** 작성한다. `CLAUDE.md`는 `@AGENTS.md` 포인터이므로 절대 수정하지 않는다.
2. AGENTS.md는 **100줄 이내**로 유지한다. (연구: 150줄 초과 시 성능 개선 없이 비용만 증가)
3. "매 턴 적용되는 행동 규칙"만 AGENTS.md에 넣는다. 특정 작업에서만 쓰는 절차는 스킬로 만든다.
4. 기존 규칙과 겹치면 합치고, 어긋나면 사용자에게 어느 쪽이 맞는지 확인한다.

## 품질 규칙 (연구 근거)

- **일반론 금지.** "클린 코드를 쓴다" 같은 에이전트가 이미 아는 말은 넣지 않는다.
  이 프로젝트 특유의, 말해주지 않으면 모를 규칙만 넣는다. (일반론은 오히려 성능을 떨어뜨림)
- **금지 나열 과다 금지.** 금지가 쌓이면 에이전트가 소극적으로 변한다.
  새 제약은 `## 경계` 표의 3열(항상/먼저 묻기/금지) 균형을 유지하며 넣는다.
- **명령은 실행 가능하게.** 스택 확정 후 명령어는 플래그까지 정확히 적는다.
  (명시된 도구는 사용률이 수십 배 높아짐)

## 실수 노트 관리

- AGENTS.md의 `## 실수 노트`는 사용자·AI 둘 다 승인 없이 추가할 수 있는 유일한 섹션.
  AI는 같은 실수가 2회 반복되면 한 줄 추가하고 그 사실을 보고한다.
- 항목이 10개를 넘으면 정리한다: 비슷한 항목 통합 → 무효 항목 삭제 → 일반 규칙으로
  승격할 것은 해당 섹션·경계 표로 이동.

## 새 스킬 추가 절차

1. 정본은 `.agents/skills/<이름>/SKILL.md`에 작성한다 (Codex·Antigravity 네이티브 인식).
2. Claude Code용 심링크: `ln -s ../../.agents/skills/<이름> .claude/skills/<이름>`
3. AGENTS.md `## 스킬` 트리거 표에 행을 추가한다. 트리거는 "요청받을 때"가 아니라
   "~하게 될 때"처럼 상황 기준으로 쓴다 (명시 요청 + 작업 흐름 모두 커버).
4. description은 3인칭 한 문장으로 "무엇을 + 언제"를 쓴다. 사용자가 실제 쓰는 단어(커밋, PR 등)를
   문장 앞쪽에 자연스럽게 넣고, 별도 키워드 나열은 하지 않는다 (길면 뒤부터 잘려 오히려 손실).
   다른 스킬과 의미가 겹치지 않게 쓴다. 발동의 최종 보증은 AGENTS.md `## 스킬`의 트리거 표다.
5. SKILL.md 본문은 간결하게. 길어지면 참조 파일로 분리한다 (progressive disclosure).

> 알려진 이슈: Claude Code가 심링크 스킬을 인식 못 하는 회귀가 간헐 보고됨.
> 스킬이 자동 트리거되지 않으면 심링크를 실제 디렉토리 복사로 전환한다:
> `rm .claude/skills/<이름> && cp -r .agents/skills/<이름> .claude/skills/` (이후 정본 수정 시 재복사)

## 프로젝트 정보 템플릿

주제가 확정되면 AGENTS.md의 `## 프로젝트 정보` 섹션을 아래로 교체하고 실제 값을 채운다.
**(미정) 항목은 채우기 전까지 추가하지 않는다** — 빈 플레이스홀더는 컨텍스트 낭비.

```markdown
## 프로젝트 정보

- 주제/목표:
- 대상 사용자:
- 스택: <언어> / <프레임워크> / <DB> / <배포>

### 명령어 (플래그까지 정확히)
- 개발 서버:
- 테스트:
- 린트/포맷:

### 고유 규칙
- <팀 컨벤션, API 규칙, 금지 사항>
```

디렉토리 구조는 코드가 곧 문서이므로 적지 않는다. 비직관적인 부분만 한 줄 메모.

## 확장 가이드 (프로젝트가 커질 때)

| 상황 | 위치 |
|---|---|
| 매 턴 적용되는 핵심 규칙 | `AGENTS.md` (100줄 상한) |
| 특정 작업의 반복 절차 | `.agents/skills/<이름>/SKILL.md` + 심링크 |
| 영역별 세부 규칙 (api, frontend 등) | `.claude/rules/<영역>.md` |
| 모노레포 디렉토리별 규칙 | `backend/AGENTS.md` 등 하위 디렉토리 |

분리 시점: 해당 내용이 "특정 작업에서만 필요"해지는 순간. 미리 빈 파일을 만들지 않는다.
