# 0015. 충전과 주문에 멱등키 적용

- 상태: 승인됨
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

네트워크 응답이 유실되면 클라이언트는 충전이나 주문의 성공 여부를 알 수 없어 요청을 재시도한다. 같은 변경이 여러 번 반영되면 포인트와 주문 데이터가 훼손되므로 여러 애플리케이션 인스턴스에서도 하나의 논리 요청을 식별해야 한다. [ADR 0002](./0002-protect-mutations-with-idempotency-keys.md)의 멱등 전략은 유지하되, 임의의 포인트 상한을 제거한 현재 계약에 맞춰 저장할 비즈니스 실패를 다시 정의할 필요가 있다.

## 고려한 대안

- 클라이언트 재시도 금지 — 서버 구현이 단순함 / 응답 유실을 안전하게 복구할 수 없음
- DB 유일 제약만 적용 — 중복 쓰기를 막음 / 최초 응답 재사용과 같은 키의 다른 요청 판별이 어려움
- 멱등 레코드를 별도 트랜잭션으로 선점 — 진행 상태를 관찰할 수 있음 / 중단된 상태의 lease와 복구 절차가 추가됨
- 업무 변경과 멱등 결과를 한 트랜잭션으로 저장 — 원자성이 단순함 / 동일 키 요청이 최초 트랜잭션 완료를 기다림

## 결정

포인트 충전과 주문 API에 UUID 문자열 형식의 `Idempotency-Key`를 필수로 요구한다. 멱등 범위는 `(user_id, operation_type, idempotency_key)`이며 DB UNIQUE 제약으로 하나의 요청만 선점한다. 요청 내용은 `CHARGE|<amount>` 또는 `ORDER|<menuId>`의 UTF-8 SHA-256 해시로 비교한다.

멱등 레코드와 업무 처리는 하나의 DB 트랜잭션에서 수행한다. 동일 키·동일 요청은 최초 HTTP 상태와 응답 본문을 재사용하고, 동일 키·다른 요청은 `409 IDEMPOTENCY_KEY_REUSED`로 거절한다. 성공과 메뉴 부재·포인트 부족·포인트 덧셈 overflow 같은 예상 가능한 비즈니스 실패는 `COMPLETED` 결과로 저장한다. 인증 실패, 멱등키·JSON·필드 형식 오류와 유효하지 않은 충전금액은 트랜잭션 전에 거절하고 저장하지 않는다. DB 오류, 락 timeout과 deadlock은 전체를 롤백하고 같은 키의 재시도를 허용한다.

`COMPLETED` 레코드는 `completed_at`부터 최소 24시간 보존하고 이후 삭제할 수 있다. 자동 정리 시각은 보장하지 않으며 실제로 삭제된 키만 새 요청으로 처리될 수 있다.

## 결과

- 응답 유실과 동시 재시도에도 충전·차감·주문은 한 번만 반영된다.
- 임의의 포인트 상한 없이도 기술적 overflow 실패를 반복 요청에서 동일하게 재현한다.
- 단일 트랜잭션이므로 중단된 `PROCESSING` 레코드용 별도 lease가 필요하지 않다.
- 동일 키의 동시 요청은 최초 트랜잭션 완료를 기다리므로 유일 키와 사용자 행의 경합을 관찰해야 한다.
- 이 ADR은 ADR 0002를 대체한다.
