# 0009. 단일 메뉴 주문과 주문 시점 스냅샷 사용

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

현재 주문 요구사항은 사용자가 메뉴 하나를 선택해 포인트로 즉시 결제하는 흐름이다. 장바구니, 수량, 옵션, 부분 취소와 환불은 범위 밖이다. 메뉴 정보가 나중에 바뀌더라도 과거 주문의 이름과 실제 결제금액은 주문 당시 값으로 설명할 수 있어야 한다.

## 고려한 대안

- 주문 헤더와 `order_items` — 다품목·수량 확장에 유리함 / 현재 한 메뉴 주문에는 조인과 모델이 불필요하게 늘어남
- 주문에서 현재 메뉴만 참조 — 데이터 중복이 없음 / 메뉴 변경 뒤 과거 주문의 표시와 결제 근거가 달라짐
- 모든 메뉴 속성 복제 — 과거 화면을 완전히 재현할 수 있음 / 현재 요구에 필요 없는 중복과 변경 관리가 늘어남
- 단일 주문 행에 핵심 스냅샷 저장 — 현재 범위가 단순하고 결제 근거를 보존함 / 다품목 확장 시 마이그레이션이 필요함

## 결정

주문 하나는 메뉴 하나와 수량 1을 의미하며 별도 `order_items` 테이블을 만들지 않는다. `orders`는 사용자·메뉴 FK와 함께 주문 시점의 메뉴명 `menu_name_snapshot`, 실제 결제한 정수 포인트 `paid_amount`, 결제 시각을 보존한다. 현재 생성되는 주문은 포인트 차감이 함께 확정된 `PAID` 상태뿐이다.

메뉴의 현재 이름·가격과 과거 주문 스냅샷의 책임을 구분한다. 주문 상세와 외부 이벤트의 결제금액은 스냅샷을 사용하고, 인기 메뉴 응답의 표시 이름·가격은 [ADR 0004](./0004-calculate-popular-menus-from-paid-orders.md)에 따라 현재 메뉴를 사용한다.

## 결과

- 현재 주문 트랜잭션과 조회 모델이 단순하고 당시 결제 근거가 보존된다.
- 메뉴 가격이나 이름이 바뀌어도 과거 주문의 결제금액과 표시 이름은 변하지 않는다.
- 수량·다품목·부분 취소를 지원할 수 없으며 요구가 생기면 `orders`를 헤더로 전환하고 `order_items`를 도입해야 한다.
- 전환 시 기존 단일 주문을 수량 1인 항목으로 이관하고 API 계약도 새 ADR로 갱신해야 한다.
