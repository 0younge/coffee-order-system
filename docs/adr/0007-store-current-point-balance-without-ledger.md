# 0007. 포인트 원장 없이 현재 잔액 저장

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

현재 요구사항은 사용자가 포인트를 충전하고 단일 메뉴 주문에 사용하는 흐름이다. 환불, 만료, 관리자 조정, 거래 감사와 잔액 재구성은 범위에 포함되지 않는다. 원장 모델은 이력을 보존하지만 모든 증감 유형과 보상 정책을 함께 설계해야 하므로 현재 범위를 크게 넓힌다.

## 고려한 대안

- 불변 포인트 원장과 합계 집계 — 감사와 잔액 재구성이 가능함 / 거래 유형·보상·집계 성능 설계가 추가됨
- 이벤트 소싱 — 모든 상태 변화를 보존함 / 저장·재생·버전 관리 복잡성이 현재 요구보다 큼
- 별도 Wallet 엔터티 — 향후 여러 자산을 확장하기 쉬움 / 사용자당 포인트 하나인 현재 모델에 불필요한 간접 계층임
- 사용자 행의 현재 잔액 — 단순하고 잠금 경계가 명확함 / 증감 이력을 독립적으로 감사할 수 없음

## 결정

`users.point_balance`에 사용자의 현재 사용 가능 포인트를 정수로 저장한다. 신규 사용자는 0P로 시작하고 1원은 1P이며 잔액은 0P 이상 10,000,000P 이하여야 한다. 한 번에 충전할 수 있는 금액은 최대 1,000,000P다. 충전과 주문은 [ADR 0001](./0001-use-database-pessimistic-locking-for-points.md)의 사용자 행 비관적 락과 DB 트랜잭션을 통해 잔액을 변경한다.

주문에는 실제 차감한 `paid_amount`를 보존해 해당 주문의 결제금액을 확인할 수 있지만 충전·차감 전체를 나타내는 별도 포인트 원장은 만들지 않는다. 포인트 수정 API, 환불, 만료와 관리자 조정도 현재 범위에 포함하지 않는다.

## 결과

- 잔액 조회와 잠금 경계가 사용자 한 행으로 단순해진다.
- `point_balance >= 0` 제약과 동시성 테스트로 핵심 불변식을 직접 검증할 수 있다.
- 충전과 주문 밖의 잔액 변화 이력을 완전하게 감사하거나 잔액을 원장에서 재계산할 수 없다.
- 환불·만료·감사 요구가 생기면 기존 잔액을 초기 원장으로 이관하고 증감 유형과 보상 거래를 새 ADR로 설계해야 한다.
