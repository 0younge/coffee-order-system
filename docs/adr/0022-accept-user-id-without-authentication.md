# 0022. 인증 없이 요청 본문의 사용자 ID 사용

- 상태: 승인됨
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

과제 원문은 포인트 충전과 주문에서 사용자 식별값을 입력받도록 요구하지만 회원가입, 로그인과 인증 체계는 요구하지 않는다. 사용자 식별값을 인증 주체로 확장해 해석한 기존 결정은 개인적인 해석을 요구사항처럼 반영한 과한 설계였다. JWT 기반 인증은 사용자 식별을 안전하게 만들 수 있지만 현재 과제 범위보다 API, 데이터 모델, 설정과 테스트를 크게 늘리므로 요구사항에 직접 대응하는 단순한 사용자 식별 계약으로 되돌릴 필요가 있다.

## 고려한 대안

- 회원가입·로그인과 JWT 유지 — 사용자 소유권을 검증할 수 있음 / 과제 원문에 없는 인증 기능과 운영 정책이 추가됨
- 별도 사용자 생성 API 없이 API Key나 고정 토큰 사용 — 호출자를 제한할 수 있음 / 사용자 식별 요구와 무관한 보안 계약이 추가됨
- 요청 본문에서 기존 사용자 ID 입력 — 원문 요구와 직접 대응하고 구현이 단순함 / 호출자가 다른 사용자 ID를 지정하는 것을 막지 않음

## 결정

회원가입, 로그인, 비밀번호와 JWT 인증은 현재 범위에서 제외한다. 포인트 충전 요청은 `userId`와 `amount`, 주문 요청은 `userId`와 `menuId`를 JSON 본문으로 받는다. `userId`는 signed `BIGINT` 범위의 양의 정수여야 하며 존재하는 `users.id`를 가리켜야 한다.

사용자 생성과 관리 API는 제공하지 않는다. 사용자는 시스템에 이미 존재한다고 가정하고 테스트는 필요한 사용자를 fixture로 준비한다. 존재하지 않는 사용자는 상태를 변경하지 않고 `404 USER_NOT_FOUND`로 거절하며 멱등 결과를 저장하지 않는다. 충전과 주문의 멱등 범위는 계속 `(user_id, operation_type, idempotency_key)`를 사용한다.

Spring Security, OAuth2 Resource Server, BCrypt, JWT 서명 키와 인증 전용 설정은 목표 구현에서 사용하지 않는다. 다중 인스턴스 정합성은 인증 상태가 아니라 공용 MySQL, 사용자 행 비관적 락, 멱등 레코드와 Transactional Outbox로 보장한다.

이 결정은 ADR 0008, 0012, 0021을 대체한다. ADR 0005의 Spring Security 기준선, ADR 0006의 Auth 모듈, ADR 0015의 인증 실패 예외, ADR 0018의 공유 JWT 키 조건도 이 결정 이후 적용하지 않으며 각 ADR의 나머지 결정은 유지한다.

## 결과

- API가 과제 원문의 사용자 식별값 입력 계약에 직접 대응한다.
- 회원가입·로그인·토큰 발급·검증과 인증 데이터가 제거되어 구현과 테스트 범위가 줄어든다.
- 호출자가 지정한 사용자 ID의 소유권은 검증하지 않으므로 실제 서비스에서 사용하려면 별도 인증·인가 결정을 새 ADR로 추가해야 한다.
- 기존 사용자를 준비하는 운영 절차는 현재 범위 밖이며 테스트는 독립 fixture를 사용한다.
