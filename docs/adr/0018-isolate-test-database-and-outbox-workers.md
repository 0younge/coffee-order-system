# 0018. 테스트 데이터베이스와 Outbox 워커 격리

- 상태: 승인됨
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

실제 MySQL을 사용하는 통합 테스트와 개발 애플리케이션이 같은 데이터베이스를 공유하면 행 식별자를 고유하게 만들어도 비동기 Outbox 워커까지 격리되지는 않는다. 워커는 테스트 소유 여부와 관계없이 전송 가능한 전체 이벤트를 조회하므로 개발 이벤트를 테스트 워커가 선점하거나 테스트 이벤트를 개발 워커가 전송할 수 있다. MySQL 제품과 Compose 실행 방식은 유지하면서 데이터와 비동기 실행 경계를 분리해야 한다.

## 고려한 대안

- 개발·테스트가 같은 데이터베이스 사용 — 설정이 가장 적음 / 전역 Outbox 조회와 병렬 실행이 서로 간섭할 수 있음
- 테스트용 MySQL 서비스를 별도 실행 — 프로세스와 볼륨까지 완전히 격리됨 / 컨테이너·포트·운영 설정이 늘어남
- 같은 Compose MySQL 안에서 데이터베이스 분리 — 실제 MySQL 의미와 단일 컨테이너를 유지함 / 테스트 데이터베이스 초기화와 접속 설정이 추가됨

## 결정

하나의 Compose MySQL 인스턴스 안에 개발용 `coffee_order_system`과 테스트용 `coffee_order_system_test`를 분리한다. Compose 초기화 과정은 두 데이터베이스와 애플리케이션 사용자의 접근 권한을 준비하며, 테스트 프로필은 반드시 테스트 데이터베이스 URL을 사용한다. 기존 볼륨에 테스트 데이터베이스가 없을 때는 명시적인 초기화 절차를 수행하고 볼륨이나 개발 데이터를 자동 삭제하지 않는다.

일반 테스트에서는 Outbox 워커를 비활성화한다. Outbox 전달·lease·fencing을 검증하는 전용 테스트만 테스트 프로세스가 제어하는 Mock HTTP 서버와 함께 워커를 활성화한다. 다중 인스턴스 검증은 한 JUnit 프로세스에서 서로 다른 랜덤 포트의 독립 Spring Context 두 개를 실행하되 같은 테스트 데이터베이스와 JWT 키를 사용하고 Outbox 워커는 끈다.

## 결과

- 테스트 워커와 개발 워커가 서로의 이벤트를 선점하지 않는다.
- 실제 MySQL 8.4의 락·트랜잭션·제약 검증과 단일 Compose MySQL 운영은 유지된다.
- 테스트 데이터베이스 초기화와 프로필 설정을 별도로 관리해야 한다.
- 테스트는 여전히 자신이 만든 행만 FK 순서로 정리하며 전체 `TRUNCATE`, schema 삭제와 `Flyway clean`을 사용하지 않는다.
