# 0021. 무상태 Bearer REST 보안 경계 구성

- 상태: 대체됨 (→ 0022)
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

JWT Access Token과 Spring Security 사용은 이미 결정했지만 Spring Security의 기본 세션·로그인·CSRF 동작을 그대로 두면 Bearer 기반 POST API 계약과 충돌할 수 있다. 프론트엔드와 브라우저 쿠키 인증은 현재 범위가 아니므로 공개 경로와 보호 경로를 명시한 무상태 REST 경계가 필요하다.

## 고려한 대안

- Spring Security 기본 동작 유지 — 초기 설정이 적음 / form login·세션·CSRF가 REST API 응답 계약과 충돌할 수 있음
- 무상태 Bearer 경계와 모든 Origin CORS 허용 — 브라우저 호출이 쉬움 / 현재 범위에 불필요하고 과도한 교차 출처 허용이 됨
- 무상태 Bearer 경계만 구성 — JWT 계약과 다중 인스턴스 목표에 맞음 / 브라우저 교차 출처 지원이 필요해지면 별도 정책이 필요함

## 결정

Spring Security를 무상태 Bearer REST API로 구성한다. 서버 인증 세션을 만들지 않고 CSRF, form login과 HTTP Basic을 비활성화한다. 회원가입, 로그인, 메뉴 목록, 인기 메뉴와 상세 정보를 숨긴 health endpoint는 공개하고 포인트 충전과 주문은 유효한 JWT 인증을 요구한다.

현재 프론트엔드가 범위 밖이므로 CORS 설정을 추가하지 않는다. 인증 실패와 보호 API 오류는 API 공통 응답 계약으로 변환하고, JWT 전체 문자열이나 내부 검증 예외를 응답과 로그에 노출하지 않는다.

## 결과

- POST API가 쿠키·세션·CSRF 토큰 없이 Bearer 인증 계약으로 일관되게 동작한다.
- 여러 애플리케이션 인스턴스가 서버 로컬 인증 상태 없이 같은 요청을 처리할 수 있다.
- 브라우저 교차 출처 호출이 필요해지면 허용 Origin과 자격 증명 정책을 별도로 결정해야 한다.
- 공개 경로를 추가할 때 Security 설정과 API 명세·인수 테스트를 함께 갱신해야 한다.
