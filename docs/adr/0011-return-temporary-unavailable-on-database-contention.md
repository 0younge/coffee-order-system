# 0011. DB 경합 timeout과 deadlock에 일시적 오류 반환

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

사용자 행 비관적 락과 멱등키 UNIQUE 경합은 정상적인 동시성 제어지만, 무제한 대기는 API 응답을 장시간 점유할 수 있다. deadlock의 희생 트랜잭션이나 락 대기 timeout은 비즈니스 실패가 아니며 같은 논리 요청을 안전하게 다시 시도할 수 있어야 한다.

## 고려한 대안

- DB 기본 timeout까지 대기 — 별도 정책이 필요 없음 / 클라이언트 대기 시간이 지나치게 길어질 수 있음
- `500 INTERNAL_SERVER_ERROR` — 기존 오류 코드 재사용 가능 / 일시적이고 재시도 가능한 성격이 드러나지 않음
- 애플리케이션에서 즉시 자동 재시도 — 클라이언트에 투명함 / 경합이 심할 때 트랜잭션 부하와 응답 시간이 늘어남
- 5초 뒤 `503` 반환 — 재시도 가능성이 명확함 / 클라이언트가 같은 멱등키로 재시도해야 함

## 결정

사용자 행 또는 멱등키 경합의 DB 락 대기는 최대 5초로 제한한다. 대기가 5초를 넘거나 deadlock의 희생 트랜잭션이 되면 현재 트랜잭션을 전체 롤백하고 `503 TEMPORARILY_UNAVAILABLE`을 반환한다. 이 결과는 멱등 레코드에 저장하지 않으며 클라이언트는 같은 `Idempotency-Key`로 재시도할 수 있다.

## 결과

- 경합 중인 요청이 무제한 대기하지 않고 재시도 가능한 공개 오류를 받는다.
- 롤백된 요청에는 업무 변경과 멱등 결과가 남지 않는다.
- 동일 멱등키의 최초 트랜잭션이 이미 커밋됐을 가능성이 있으므로 재시도는 반드시 같은 키를 사용해야 한다.
- timeout과 deadlock 발생률을 관찰해 5초 기준과 경합 원인을 재검토해야 한다.
