# 0024. 과제와 로컬 실행용 기준 사용자 seed

- 상태: 승인됨
- 날짜: 2026-07-16
- 결정자: 프로젝트 소유자

## 맥락

충전과 주문은 기존 `userId`를 입력받지만 사용자 생성·관리 API는 범위에서 제외돼 있다. 테스트는 독립 fixture로 사용자를 준비할 수 있으나, 빈 로컬 DB에는 유효한 사용자가 없어 수동 API 검증을 바로 수행할 수 없다. 사용자 관리 기능을 추가하지 않으면서 과제 제출과 로컬 실행의 재현 가능한 시작점을 제공할 필요가 있다.

## 고려한 대안

- 사용자 생성 API 추가 — 수동 준비가 쉬움 / 원문에 없는 회원·사용자 관리 기능과 계약이 추가됨
- README에 수동 INSERT만 제공 — 제품 데이터와 fixture가 분리됨 / 실행자마다 별도 준비가 필요하고 빈 DB 재현성이 낮아짐
- Flyway로 기준 사용자 1명 seed — 빈 DB에서 바로 충전·주문 가능 / 업무 사용자 데이터와 과제용 기준 데이터가 함께 배포됨

## 결정

Flyway는 과제·로컬 수동 검증용 기준 사용자 `id=1`, `point_balance=0`을 초기 데이터로 저장한다. 사용자 생성·관리 API, 회원가입, 로그인, 인증과 사용자 ID 소유권 검증은 계속 구현하지 않는다.

자동화 테스트는 기준 사용자에 의존하거나 그 행을 변경하지 않고 테스트별 독립 fixture를 생성·정리한다. 기준 사용자는 과제와 로컬 실행의 재현 가능한 입력일 뿐 실제 서비스의 사용자 프로비저닝 정책을 뜻하지 않는다.

이 결정은 ADR 0022의 “기존 사용자를 준비하는 운영 절차는 현재 범위 밖”이라는 결과만 부분 대체하며, 요청 본문의 `userId` 사용과 무인증 경계 등 나머지 결정은 유지한다.

## 결과

- 빈 DB에서도 `userId=1`을 충전한 뒤 주문하는 수동 시나리오를 즉시 실행할 수 있다.
- 테스트 데이터 독립성과 사용자 생성 API 제외 범위는 유지된다.
- 실제 운영으로 확장할 때는 기준 seed를 제거하고 사용자 프로비저닝과 인증·인가를 새 ADR로 결정해야 한다.
