# 0017. Outbox 최초 전송 시도 시간을 제한

- 상태: 승인됨
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

주문 내역을 데이터 수집 플랫폼으로 실시간 전송해야 하지만 고객 주문 트랜잭션에서 외부 HTTP를 호출하면 외부 장애가 주문 성공과 DB 락 시간에 전파된다. ADR 0003과 0014는 Transactional Outbox와 1초 폴링을 선택했지만, “가능한 즉시”만으로는 실시간 요구사항을 자동화 테스트로 판정할 수 없다.

## 고려한 대안

- 주문 트랜잭션 안에서 동기 전송 — 요청 시점 전달이 명확함 / 외부 장애가 주문 원자성과 응답 지연에 영향을 줌
- 커밋 후 메모리 알림만 사용 — 평균 지연이 짧음 / 프로세스 중단 시 알림이 유실될 수 있음
- 1초 Outbox 폴링에 최초 시도 기준 추가 — 영속성과 장애 격리를 유지함 / backlog와 워커 장애 시에는 기준을 보장할 수 없음

## 결정

주문 커밋과 같은 트랜잭션에서 Outbox 이벤트의 `next_retry_at`을 즉시 전송 가능한 시각으로 저장한다. 워커가 정상이고 전송 가능한 기존 backlog가 없는 조건에서 주문 커밋 후 2초 이내에 해당 이벤트의 최초 외부 HTTP 요청을 시작한다. 2초 기준은 외부 시스템의 응답 완료나 `PUBLISHED` 전환 시간이 아니라 요청 시작 시각에 적용한다.

주문 API는 외부 응답을 기다리지 않는다. 기본 폴링 주기 1초와 인스턴스당 활성 배치 1개 정책을 유지하고, 테스트는 고정 sleep이 아니라 제한 시간 안에서 Mock API의 수신 기록을 확인하는 bounded polling을 사용한다. backlog·워커 중단·외부 장애에서는 발행 지연과 oldest pending age를 관측하고 복구·재시도 계약을 우선한다.

## 결과

- “실시간”이 주문과 외부 장애를 분리한 준실시간 최초 시도 계약으로 측정 가능해진다.
- 정상 조건의 최초 전송 지연 회귀를 자동화 테스트로 잡을 수 있다.
- 2초 안에 외부 성공 응답이나 최종 발행을 보장하지는 않는다.
- backlog가 있는 부하 조건의 허용 지연은 구현 후 별도 성능 기준으로 정한다.
