# 0016. 임의 상한 없이 현재 포인트 잔액 저장

- 상태: 승인됨
- 날짜: 2026-07-15
- 결정자: 프로젝트 소유자

## 맥락

현재 요구사항은 사용자가 양의 정수 포인트를 충전하고 단일 메뉴 주문에 사용하는 흐름이다. 환불·만료·관리자 조정·거래 감사는 범위 밖이므로 포인트 원장보다 현재 잔액 모델이 단순하다. [ADR 0007](./0007-store-current-point-balance-without-ledger.md)의 1회 충전 1,000,000P와 잔액 10,000,000P 제한은 원문 요구사항에 없는 임의 정책이므로 제거한다.

## 고려한 대안

- 기존 업무 상한 유지 — 입력과 저장 범위가 작고 명확함 / 근거 없는 제한이 정상 충전 요청을 거절할 수 있음
- 불변 포인트 원장 — 감사와 잔액 재구성이 가능함 / 현재 요구보다 데이터 모델과 보상 정책이 커짐
- 사용자 행의 현재 잔액과 양수 규칙만 유지 — 잠금 경계가 단순하고 과제 계약에 직접 대응함 / 증감 이력을 재구성할 수 없음

## 결정

`users.point_balance`에 현재 사용 가능한 포인트를 MySQL signed `BIGINT`로 저장한다. 신규 사용자는 0P로 시작하며 DB는 `CHECK (point_balance >= 0)`로 음수 잔액을 막는다. 충전 `amount`는 signed `BIGINT` 범위의 0보다 큰 정수만 허용하고 임의의 1회 충전 상한이나 잔액 상한은 두지 않는다.

충전 전에 현재 잔액과 `amount`의 덧셈이 signed `BIGINT` 범위를 넘는지 검사한다. 입력 자체가 범위를 벗어나면 `400 INVALID_CHARGE_AMOUNT`, 유효한 입력이 기존 잔액과 합산될 때 overflow가 발생하면 상태를 변경하지 않고 `409 POINT_BALANCE_OVERFLOW`를 반환한다. 충전과 주문은 사용자 행 비관적 락과 DB 트랜잭션으로 잔액을 변경한다.

주문에는 실제 차감한 `paid_amount`를 보존하지만 별도 포인트 원장, 수정 API, 환불, 만료와 관리자 조정은 만들지 않는다.

## 결과

- 원문에 없는 금액 상한 때문에 정상 요청이 거절되지 않는다.
- 잔액 조회와 동시성 제어는 사용자 한 행으로 유지된다.
- signed `BIGINT`의 기술적 범위와 덧셈 overflow는 API와 테스트에서 명시적으로 처리한다.
- 충전·차감 이력을 독립적으로 감사하거나 잔액을 원장에서 재계산할 수 없다.
- 이 ADR은 ADR 0007을 대체한다.
