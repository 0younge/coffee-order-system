# 0003. 주문 이벤트를 Transactional Outbox로 전달

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

주문 트랜잭션 안에서 외부 수집 API를 호출하면 외부 장애와 지연이 고객 주문에 전파된다. 반대로 주문 커밋 뒤 단순 비동기 호출만 수행하면 프로세스 중단 시 주문은 남고 이벤트는 유실될 수 있다. 여러 워커가 동시에 동작하고 처리 lease가 만료되더라도 오래된 워커가 새 처리 결과를 덮어쓰지 않게 해야 한다.

## 고려한 대안

- 주문 트랜잭션 안에서 동기 HTTP 호출 — 흐름은 직관적임 / 외부 장애가 DB 락 시간과 주문 성공 여부에 영향을 줌
- 커밋 후 메모리 큐나 스레드로 전송 — 구현은 단순함 / 프로세스 중단 시 이벤트를 복구할 영속 기록이 없음
- Kafka에 직접 발행 — 높은 처리량과 독립 소비자를 지원함 / DB와 브로커의 원자적 이중 쓰기와 운영 복잡성이 남음
- Transactional Outbox — 기존 MySQL 트랜잭션으로 주문과 이벤트를 함께 보존함 / 폴링·재시도·정리 워커를 운영해야 함

## 결정

포인트 차감, 주문 저장, Outbox 이벤트 저장을 하나의 MySQL 트랜잭션으로 커밋한다. 주문 하나에는 고유한 `ORDER_PAID` 이벤트 하나만 만들고 외부 요청마다 같은 `eventId`를 전송한다. 전달 보장은 at-least-once이며 수신 측은 `eventId`로 중복 반영을 막는다.

워커는 MySQL 8.4의 `FOR UPDATE SKIP LOCKED`로 처리 가능한 행을 짧게 잠그고 다음 값을 기록한 뒤 커밋한다.

- 상태: `PROCESSING`
- `claim_token`: 점유 시도를 고유하게 식별하는 불투명 토큰
- `locked_at`: lease 시작 시각

외부 HTTP 호출은 DB 트랜잭션 밖에서 수행한다. 성공·실패 상태 갱신은 `event_id`와 `claim_token`이 모두 일치할 때만 허용한다. 30초 lease가 만료된 이벤트를 다른 워커가 새 토큰으로 재점유하면 이전 워커의 늦은 응답은 상태를 변경하지 못한다.

워커는 기본 1초마다 최대 50건을 `next_retry_at`, `created_at`, `event_id` 오름차순으로 선점한다. 주기와 배치 크기는 환경 설정으로 바꿀 수 있다.

전송 정책은 다음과 같다.

- 연결 timeout 2초, 응답 timeout 5초
- 최초 시도는 커밋 후 즉시, 실패 시 1분·5분·30분 뒤 재시도하여 총 4회 시도
- HTTP `2xx`는 성공으로 처리하고 `PUBLISHED`로 전환
- 연결 오류, timeout, `408`, `429`, `5xx`는 다음 시각을 예약해 재시도
- 그 밖의 `4xx` 또는 네 번째 시도 실패는 `FAILED`로 전환
- redirect를 따르지 않으며 `3xx`는 즉시 `FAILED`로 전환

외부 API의 base URL은 기본값 없는 `COLLECTION_API_BASE_URL` 환경 변수로 주입하고 인증은 현재 범위에서 사용하지 않는다. `POST /events/orders`의 HTTP 상태만 성공 여부에 사용하며 응답 본문은 무시한다.

`PUBLISHED` 이벤트는 발행 완료 후 최소 30일 보존하고 이후 삭제할 수 있지만 자동 정리 스케줄러는 현재 범위에 포함하지 않는다. `FAILED` 이벤트에는 `failed_at`, 선택적인 `last_http_status`, `last_error_type`만 저장하고 응답 본문이나 stack trace는 저장하지 않는다. `FAILED` 이벤트는 자동 삭제하지 않으며 수동 재시도 API도 현재 범위에 포함하지 않는다.

## 결과

- 고객 주문 성공은 외부 API의 가용성과 분리되고 커밋된 이벤트는 프로세스 재시작 뒤에도 복구된다.
- 여러 인스턴스가 이벤트를 나눠 처리하면서 claim token이 lease 만료 경쟁의 stale update를 막는다.
- 응답 유실과 lease 재점유로 같은 이벤트가 여러 번 전달될 수 있으므로 수신 측 중복 제거가 필수다.
- Outbox backlog, 가장 오래된 미처리 이벤트 나이, 재시도 수, `FAILED` 수와 claim 충돌을 관찰해야 한다.
- Kafka 같은 브로커로 전환하려면 전달·재처리·운영 모델과 이 ADR을 새 결정으로 대체해야 한다.
