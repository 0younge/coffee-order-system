# 0001. 포인트 변경에 DB 비관적 락 사용

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

여러 애플리케이션 인스턴스가 같은 사용자의 포인트 잔액을 동시에 읽고 변경하면 갱신 유실이나 초과 결제가 발생할 수 있다. 충전과 주문은 잔액이라는 동일한 불변식을 공유하므로 한 가지 동시성 경계로 직렬화해야 한다. 주문·포인트의 단일 진실 원천은 MySQL이며 애플리케이션 인스턴스는 로컬 상태를 공유하지 않는다.

## 고려한 대안

- JVM 로컬 락 — 구현은 단순함 / 여러 인스턴스 사이의 요청을 직렬화하지 못함
- 낙관적 락과 충돌 재시도 — 충돌이 적을 때 병렬성이 좋음 / 같은 사용자의 경쟁이 심하면 재시도와 응답 지연이 증가함
- Redis 분산 락 — 애플리케이션 수준에서 락을 공유할 수 있음 / 락 만료와 DB 트랜잭션 사이에 별도 실패 모드가 생김
- 조건부 원자 갱신 — 단순 차감에는 효율적임 / 충전·주문 스냅샷·Outbox를 함께 처리하는 흐름의 가독성과 일관된 잠금 순서를 해침

## 결정

포인트 충전과 주문 결제는 같은 DB 트랜잭션 안에서 `users` 행을 사용자 PK로 비관적 쓰기 잠금한 뒤 잔액을 변경한다. 같은 사용자의 변경은 직렬화하고 서로 다른 사용자의 변경은 독립적으로 진행한다.

잠금을 보유한 트랜잭션에는 비밀번호 해시 계산이나 외부 HTTP 호출을 포함하지 않는다. 주문 트랜잭션은 잔액 검증·차감, 주문 저장, Outbox 저장까지만 처리하고 커밋한다. `point_balance >= 0` DB 제약은 애플리케이션 검증을 대체하지 않고 최후의 방어선으로 사용한다.

멱등 요청의 중복 방지는 [ADR 0002](./0002-protect-mutations-with-idempotency-keys.md), 주문 이벤트 저장은 [ADR 0003](./0003-deliver-order-events-with-transactional-outbox.md)을 따른다.

## 결과

- 동일 사용자에 대한 충전과 결제가 순서대로 반영되어 잔액 불변식을 설명하고 검증하기 쉽다.
- 다른 사용자끼리는 같은 행을 잠그지 않으므로 사용자 단위 병렬성은 유지된다.
- 인기 사용자에게 요청이 몰리면 행 잠금 대기 시간이 병목이 될 수 있으므로 잠금 대기 시간과 트랜잭션 지연을 관찰해야 한다.
- DB 교착 상태나 잠금 타임아웃의 재시도 정책은 구현 전에 별도 운영 정책으로 정한다.
- 다른 동시성 전략으로 바꾸려면 충전·주문 저장소와 동시성 테스트를 함께 변경해야 한다.
