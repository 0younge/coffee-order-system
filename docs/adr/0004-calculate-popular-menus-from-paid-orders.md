# 0004. 인기 메뉴를 결제 완료 주문에서 직접 집계

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

인기 메뉴는 조회 시점 직전 7×24시간 동안 실제 결제가 완료된 주문 수를 정확하게 보여 줘야 한다. 이벤트 기반 사전 집계나 캐시는 조회를 빠르게 만들 수 있지만 지연·중복 소비·무효화 문제로 원본 주문과 값이 달라질 수 있다. 현재 예상 규모와 지연 목표가 확정되지 않았으므로 먼저 가장 단순하고 정확한 원본 집계를 사용한다.

## 고려한 대안

- 애플리케이션 메모리 집계 — 조회는 빠름 / 다중 인스턴스 사이에서 값이 갈리고 재시작 시 사라짐
- Redis 캐시·정렬 집합 — 빠른 조회가 가능함 / 주문 트랜잭션과 캐시 무효화의 정합성 경계가 추가됨
- Kafka 기반 사전 집계 — 대규모 읽기 부하를 분리할 수 있음 / 소비 지연·중복 처리·운영 복잡성을 감수해야 함
- 결제 완료 주문 직접 집계 — 최신 원본과 정확히 일치함 / 주문량 증가에 따라 DB 집계 비용이 커질 수 있음

## 결정

MySQL의 `orders`를 주문 수의 단일 진실 원천으로 삼고, 상태가 `PAID`이며 UTC 기준 `[조회 시각 - 7일, 조회 시각)`에 속하는 주문을 `menu_id`별로 직접 집계한다. 주문 수 내림차순, 동률이면 메뉴 ID 오름차순으로 정렬해 최대 3개를 반환한다.

인기 메뉴의 주문 수는 주문 스냅샷에서 계산하지만 응답의 메뉴 이름과 가격은 현재 `menus` 행에서 가져온다. 주문 당시의 이름과 결제금액은 감사 가능한 주문 내역으로만 보존한다. 메뉴 관리 CRUD는 현재 범위 밖이므로 카탈로그 변경 정책이 생기면 이 응답 의미를 다시 검토한다.

`orders(status, paid_at, menu_id)` 인덱스로 기간 필터를 지원한다. 캐시나 별도 조회 모델은 실제 실행 계획과 지연이 병목임을 측정하고 허용 가능한 집계 지연을 합의한 뒤 도입한다.

## 결과

- API 결과가 커밋된 결제 완료 주문과 즉시 일치하며 별도 집계 데이터의 복구 절차가 필요 없다.
- 동률 정렬과 반개구간이 명확해 페이지마다 결과가 흔들리거나 경계 주문이 중복 집계되지 않는다.
- 응답은 현재 주문 가능한 메뉴 정보와 최근 주문 인기도를 함께 제공한다.
- 데이터가 증가하면 기간 내 스캔 행 수, `EXPLAIN ANALYZE`, 임시 테이블·정렬 사용 여부와 p50/p95/p99 지연을 관찰해야 한다.
- DB 직접 집계가 목표 성능을 만족하지 못할 때만 새 ADR로 캐시나 사전 집계 모델을 선택한다.
