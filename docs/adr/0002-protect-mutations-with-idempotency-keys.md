# 0002. 충전과 주문에 멱등키 적용

- 상태: 대체됨 (→ 0015)
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

서버가 처리를 마친 뒤 응답이 유실되면 클라이언트는 성공 여부를 알 수 없어 요청을 재시도한다. 포인트 충전이나 주문이 두 번 반영되면 잔액과 주문 데이터가 훼손되므로 네트워크 재시도를 하나의 논리 요청으로 식별해야 한다. 중복 방지는 여러 애플리케이션 인스턴스에서도 동일하게 동작해야 한다.

## 고려한 대안

- 클라이언트 재시도 금지 — 서버 구현은 단순함 / 응답 유실을 복구할 수 없고 신뢰할 수 없는 클라이언트에 의존함
- 요청 시각이나 요청 본문만으로 중복 추정 — 별도 키가 필요 없음 / 서로 다른 정상 요청을 오인하거나 동일 요청을 놓칠 수 있음
- DB 유니크 제약만 적용 — 중복 쓰기는 막을 수 있음 / 최초 응답 재사용과 같은 키의 다른 요청 판별이 어려움
- 멱등키를 별도 트랜잭션으로 먼저 확정 — 진행 상태를 다른 트랜잭션에서 볼 수 있음 / 중단된 `PROCESSING` 레코드의 lease와 복구 절차가 추가됨

## 결정

포인트 충전과 주문 API에 UUID 문자열 형식의 `Idempotency-Key`를 필수로 요구한다. 멱등 범위는 `(user_id, operation_type, idempotency_key)`이며 DB 유니크 제약으로 한 요청만 선점한다. 요청 내용은 작업 유형과 검증된 업무 값(`CHARGE|<amount>`, `ORDER|<menuId>`)을 UTF-8로 직렬화한 뒤 SHA-256 해시로 비교한다. JSON 공백·필드 순서와 알 수 없는 필드는 해시 입력이 아니다.

멱등 레코드와 비즈니스 처리는 하나의 DB 트랜잭션으로 수행한다.

1. `PROCESSING` 레코드로 키를 선점한다.
2. 포인트 충전 또는 주문 처리를 수행한다.
3. 성공이나 예상 가능한 비즈니스 실패의 HTTP 상태·결과 코드·응답 본문을 저장하고 `COMPLETED`로 바꾼 뒤 커밋한다.
4. 예상하지 못한 DB·프로세스 오류는 전체 트랜잭션과 멱등 레코드를 롤백해 같은 키의 재시도를 허용한다.

동일 키·동일 요청은 최초 확정 결과를 그대로 반환하고, 동일 키·다른 요청은 `409 IDEMPOTENCY_KEY_REUSED`로 거절한다. 동시에 도착한 동일 요청은 최초 트랜잭션의 완료를 기다린 뒤 같은 결과를 사용한다. DB 락 대기가 5초를 넘거나 deadlock의 희생 트랜잭션이 되면 전체를 롤백하고 멱등 결과를 저장하지 않은 채 `503 TEMPORARILY_UNAVAILABLE`을 반환한다.

인증 실패, 멱등키 누락·형식 오류, JSON 오류, 필수 필드 누락과 잘못된 충전 금액은 트랜잭션 전에 거절하고 멱등 결과로 저장하지 않는다. 성공과 메뉴 부재·포인트 부족·보유 잔액 상한 초과 결과만 `COMPLETED`로 저장한다.

완료 레코드는 `completed_at`부터 최소 24시간 보존하고 그 이후 삭제할 수 있다. 정확한 삭제 시각은 보장하지 않으며 현재 범위에 자동 정리 스케줄러를 두지 않는다. 실제로 정리된 키를 다시 보내면 새로운 논리 요청으로 처리될 수 있음을 API 계약에 명시한다. 응답 저장 시 JWT, 비밀번호, 자격 증명과 같은 비밀 값은 포함하지 않는다.

## 결과

- 응답 유실과 동시 재시도에도 충전·차감·주문은 한 번만 반영된다.
- 예상 가능한 실패도 반복 요청에서 동일하게 재현되어 클라이언트가 안정적으로 복구할 수 있다.
- 단일 트랜잭션이라 중단된 `PROCESSING` 레코드를 별도로 회수하는 lease가 필요하지 않다.
- 최초 처리 시간이 긴 경우 중복 요청도 DB 트랜잭션 완료를 기다리므로 대기 시간과 유니크 키 경합을 관찰해야 한다.
- 최소 24시간 이후 실제로 레코드가 삭제되면 같은 키가 새 요청으로 처리될 수 있으며 자동 정리 스케줄러는 현재 범위에 없다.
