# 0010. Docker Compose의 실제 MySQL로 통합 테스트

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

이 시스템의 핵심 위험은 사용자 행 비관적 락, 유니크·체크 제약, 트랜잭션 롤백, `SKIP LOCKED`와 집계 실행계획이다. 인메모리 DB나 저장소 mock은 MySQL의 실제 동시성·SQL 동작을 충분히 재현하지 못한다. 로컬과 자동화 환경에서 같은 DB 기준을 사용할 필요가 있다.

## 고려한 대안

- H2 같은 인메모리 DB — 빠르고 준비가 쉬움 / SQL 방언과 잠금·격리·제약 동작이 MySQL과 다름
- 저장소 mock만 사용 — 단위 테스트가 빠름 / 트랜잭션과 DB 제약을 검증하지 못함
- Testcontainers — 테스트별 격리와 자동 수명 주기가 편리함 / 현재 범위에 새 테스트 의존성과 컨테이너 시작 제어가 추가됨
- Docker Compose의 MySQL — 로컬 실행과 같은 DB를 명시적으로 공유함 / Docker 준비와 데이터 격리 절차가 필요함

## 결정

저장소·API·동시성 통합 테스트는 Docker Compose로 실행한 MySQL 8.4 LTS를 사용한다. Compose는 MySQL 서비스만 실행하고 애플리케이션과 Gradle 테스트는 호스트에서 실행한다. Flyway가 테스트 스키마와 초기 메뉴를 구성하며 테스트는 운영과 같은 MySQL 기능을 사용한다.

순수 도메인 단위 테스트는 DB 없이 빠르게 실행한다. 외부 수집 API는 mock HTTP 서버로 대체하되 구체적인 라이브러리는 구현 시 별도 승인 후 선택한다. 고정된 시간은 주입 가능한 `Clock`으로 제어하고 동시성 테스트는 서로 다른 실제 DB 커넥션과 트랜잭션을 사용한다.

## 결과

- 비관적 락, 유니크 제약, Flyway와 인기 메뉴 SQL을 실제 DB 의미로 검증할 수 있다.
- 로컬 테스트 전에 Docker와 MySQL health check가 준비돼야 하며 인메모리 테스트보다 실행 시간이 길다.
- 테스트 데이터 정리와 병렬 실행 격리를 명시적으로 관리해야 한다.
- 테스트 실행 시간이 병목이 되면 실제 DB 검증 범위를 유지한 채 컨테이너 재사용이나 Testcontainers 전환을 새 결정으로 검토한다.
