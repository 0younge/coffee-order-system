# 0006. 기능 중심 모듈러 모놀리스 사용

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

인증, 메뉴, 포인트, 주문, 멱등성, 외부 전송은 책임이 다르지만 하나의 주문 흐름과 DB 트랜잭션으로 협력한다. 현재 규모에서 서비스를 물리적으로 분리하면 분산 트랜잭션과 운영 비용이 요구사항보다 커진다. 반대로 모든 코드를 전역 계층별 패키지에 모으면 기능 경계와 변경 영향이 흐려진다.

## 고려한 대안

- 전통적인 전역 계층형 모놀리스 — 구조가 익숙함 / 기능 하나를 이해하려면 여러 최상위 패키지를 오가야 함
- 마이크로서비스 — 독립 배포와 확장에 유리함 / 현재는 분산 트랜잭션·네트워크 장애·관측성 비용이 과도함
- Gradle 멀티모듈 — 컴파일 경계를 강제할 수 있음 / 초기 규모에 비해 빌드와 의존성 관리가 복잡해짐
- 기능 중심 모듈러 모놀리스 — 단일 트랜잭션을 유지하면서 책임 경계를 드러냄 / 경계를 코드 리뷰와 테스트로 지속 관리해야 함

## 결정

애플리케이션은 하나의 프로세스와 하나의 배포 단위를 갖는 모듈러 모놀리스로 구성한다. 최상위 코드는 `Auth`, `Menu`, `Point`, `Order`, `Idempotency`, `Outbox` 기능 책임으로 나누고, 각 기능 안에서 API·Application·Domain·Infrastructure 역할을 구분한다.

다른 기능의 저장소 구현이나 내부 도메인 객체를 직접 조작하지 않고 명시적인 애플리케이션 계약을 통해 협력한다. 주문 애플리케이션 흐름은 포인트 잠금·주문·Outbox의 트랜잭션을 조정하되 각 기능의 불변식을 대신 소유하지 않는다. 패키지 경계가 충분한 현재 단계에서는 별도 Gradle 모듈을 만들지 않는다.

## 결과

- 기능과 관련 테스트를 가까이 두어 변경 범위와 책임을 찾기 쉽다.
- 포인트·주문·Outbox에 필요한 단일 DB 트랜잭션을 유지할 수 있다.
- 단일 배포 단위이므로 기능별 독립 배포는 불가능하지만 인스턴스 전체를 수평 확장할 수 있다.
- 기능 간 직접 참조가 늘면 모듈 경계가 약해질 수 있으므로 의존 방향을 구조 테스트나 리뷰로 확인해야 한다.
- 실제 독립 확장 요구가 생기면 데이터 소유권과 이벤트 계약을 먼저 정한 뒤 서비스 분리를 새 ADR로 결정한다.
