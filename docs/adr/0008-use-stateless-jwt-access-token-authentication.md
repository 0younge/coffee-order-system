# 0008. JWT Access Token 기반 무상태 인증 사용

- 상태: 대체됨 (→ 0022)
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

보호 API는 여러 애플리케이션 인스턴스에서 같은 방식으로 사용자를 식별해야 한다. 서버 로컬 세션에 의존하면 수평 확장 시 세션 공유나 고정 라우팅이 필요하다. 현재 범위에는 장기 세션, 소셜 로그인, 토큰 강제 폐기와 계정 복구가 포함되지 않는다.

## 고려한 대안

- 서버 세션 — 즉시 폐기와 중앙 통제가 쉬움 / 공유 세션 저장소나 sticky session이 필요함
- 불투명 토큰과 중앙 조회 — 토큰 정보를 통제하기 쉬움 / 매 요청마다 별도 저장소 조회가 필요함
- OAuth/OIDC 제공자 연동 — 표준화된 연합 인증을 제공함 / 현재 이메일·비밀번호 범위를 크게 넘어섬
- 짧은 수명의 JWT Access Token — 인스턴스가 독립적으로 검증 가능함 / 발급 후 만료 전 즉시 폐기가 어려움

## 결정

사용자는 이메일과 비밀번호로 가입·로그인하고 비밀번호는 BCrypt 단방향 해시로 저장한다. 이메일은 앞뒤 공백 제거 후 `@NotBlank`, `@Email`, 최대 254자로 검증하며 저장한 대소문자는 유지하되 중복 비교는 대소문자를 무시한다. 비밀번호는 최소 8자, UTF-8 기준 최대 72바이트이며 조합 규칙은 두지 않는다. 앞뒤 공백이 있으면 거절하고 중간 공백은 허용한다.

로그인 성공 시 HS256으로 서명한 JWT Access Token을 발급하며 유효기간은 30분이다. 필수 claim은 `sub` 사용자 ID, `iat`, `exp`, `iss=coffee-order-system`이다. 서명 키는 최소 32바이트 비밀값을 환경 변수로 주입한다. `jti`, `aud`, `kid`와 애플리케이션 내부 키 회전 기능은 현재 범위에 두지 않는다. 보호 API는 `Authorization: Bearer <token>`에서 인증 주체를 만들고 요청 본문의 사용자 ID를 받거나 신뢰하지 않는다.

애플리케이션은 서버 로컬 인증 세션을 보관하지 않는다. 토큰·비밀번호·서명 키를 로그에 남기지 않는다. Refresh Token, 로그아웃 토큰 목록, 강제 폐기, OAuth와 비밀번호 복구는 현재 범위 밖이다. 운영에서 무중단 키 회전이나 여러 검증 주체가 필요해지면 별도 보안 정책과 새 ADR로 결정한다.

## 결과

- 각 인스턴스가 공유 세션 없이 토큰을 검증하므로 수평 확장이 단순하다.
- 짧은 수명으로 장기 노출 위험을 제한하지만 유효한 토큰을 즉시 폐기할 수 없다.
- 인증 실패 응답은 이메일 존재 여부를 드러내지 않아야 한다.
- Refresh Token이나 강제 폐기를 도입하면 저장 모델과 위협 모델을 새 ADR에서 다시 결정해야 한다.
