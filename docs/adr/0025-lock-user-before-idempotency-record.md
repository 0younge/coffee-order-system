# 0025. 사용자 행을 멱등 레코드보다 먼저 잠금

- 상태: 승인됨
- 날짜: 2026-07-16
- 결정자: 프로젝트 소유자

## 맥락

`idempotency_records.user_id`는 `users.id`를 참조한다. MySQL InnoDB는 멱등 레코드를 삽입할 때 부모 사용자 행의 공유 락을 획득하므로, 같은 사용자의 여러 요청이 멱등 레코드를 먼저 삽입한 뒤 사용자 행의 배타 락으로 승격하면 교착 상태가 발생할 수 있다. 실제 MySQL 동시성 테스트에서 서로 다른 멱등키와 동일 멱등키 요청 모두 이 교착을 재현했다.

## 고려한 대안

- 멱등 레코드를 먼저 잠그고 deadlock을 `503`으로 반환 — 기존 흐름을 유지함 / 정상적인 동시 충전에서도 불필요한 재시도가 반복됨
- 멱등 레코드의 사용자 외래 키를 제거 — 삽입 시 부모 행 락을 피함 / 존재하지 않는 사용자와 고아 레코드를 DB가 막지 못함
- 사용자 행을 먼저 잠그고 멱등 레코드를 처리 — 같은 사용자의 요청을 일관되게 직렬화함 / 완료 응답 재사용도 사용자 행 락을 기다림

## 결정

입력 형식을 검증한 뒤 충전과 주문 트랜잭션은 사용자 행을 PK `FOR UPDATE`로 먼저 조회한다. 사용자가 없으면 멱등 레코드 없이 `USER_NOT_FOUND`로 거절한다. 사용자 행 락을 획득한 요청만 멱등 레코드를 선점·잠금하고, 완료 결과 재사용 또는 업무 변경과 결과 저장을 수행한다.

모든 충전과 주문이 사용자 행, 멱등 레코드 순서로 락을 획득한다. 멱등 레코드와 업무 결과가 한 트랜잭션에서 원자적으로 커밋되는 기존 결정은 유지한다.

## 결과

- 외래 키 확인의 공유 락에서 사용자 행 배타 락으로 승격하며 발생하던 교착을 제거한다.
- 같은 사용자의 서로 다른 변경과 멱등 재사용 요청은 사용자 행에서 직렬화된다.
- 서로 다른 사용자의 요청은 각 사용자 행에서 독립적으로 진행한다.
- 사용자 행 또는 멱등키 락이 5초를 넘거나 다른 원인의 deadlock이 발생하면 기존 정책대로 전체 롤백하고 `503`을 반환한다.
