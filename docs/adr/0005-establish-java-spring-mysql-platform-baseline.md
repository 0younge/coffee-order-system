# 0005. Java·Spring·MySQL 플랫폼 기준선 확정

- 상태: 승인됨
- 날짜: 2026-07-12
- 결정자: 프로젝트 소유자

## 맥락

초기 빌드는 Spring Boot 4.1.0으로 생성됐지만 초안 README에는 3.5.x가 적혀 있어 구현과 설계 기준이 충돌했다. 동시성·트랜잭션·실제 DB 제약을 검증하려면 런타임, 프레임워크, DB와 빌드 도구의 기준선을 하나로 고정해야 한다. 아직 구현되지 않은 구성 요소도 목표 스택과 현재 상태를 구분해 관리할 필요가 있다.

## 고려한 대안

- Spring Boot 3.5.x로 하향 — 기존 초안과 일치함 / 이미 생성된 4.1.0 빌드와 의존성을 다시 맞춰야 함
- Spring Boot 4.1.0 유지 — 현재 빌드와 일치하고 별도 마이그레이션이 없음 / 3.x 예제나 라이브러리의 호환성을 그대로 가정할 수 없음
- 인메모리 DB를 기준 DB로 사용 — 로컬 준비가 쉬움 / MySQL 잠금·격리·제약·쿼리 실행계획을 재현하지 못함

## 결정

다음 버전을 프로젝트의 목표 플랫폼 기준선으로 사용한다.

| 영역 | 기준 |
|---|---|
| Java | Amazon Corretto 17 |
| 애플리케이션 | Spring Boot 4.1.0 |
| 웹·보안·영속성 | Spring Web MVC, Spring Security, Spring Data JPA |
| 데이터베이스 | MySQL 8.4 LTS |
| 스키마 변경 | Flyway |
| 빌드 | Gradle Wrapper 9.5.1 |
| 로컬 인프라 | Docker Compose |

Docker Compose는 로컬 MySQL만 실행하고 애플리케이션은 호스트에서 `./gradlew bootRun`으로 실행한다. Spring Boot가 관리하는 의존성 버전을 우선하며, 새로운 주요 라이브러리는 호환성을 확인하고 사용자 승인 후 추가한다. 기준선에 포함됐지만 아직 코드나 설정이 없는 Flyway·Compose·JWT 업무 기능은 문서에서 구현 예정으로 표시한다.

## 결과

- README, 빌드, 테스트 전략과 실행 환경이 같은 버전 기준을 사용한다.
- MySQL 8.4의 실제 잠금과 `SKIP LOCKED` 동작을 설계·테스트할 수 있다.
- Spring Boot 3.x 전용 API나 예제를 사용할 때 4.1 호환성을 별도로 검증해야 한다.
- 프레임워크나 DB 주 버전을 바꾸려면 빌드·마이그레이션·테스트 전체를 검증하고 이 ADR을 새 ADR로 대체해야 한다.
