# 테스트 전략

## 1. 문서 목적과 현재 범위

이 문서는 구현이 완료됐다고 주장하는 결과 보고서가 아니라, 구현 과정에서 요구사항 누락과 회귀를 막기 위한 **검증 계약**이다. 테스트 결과의 현재 상태와 요구사항별 증거는 [요구사항 추적표](./requirements-traceability.md)에서 관리한다.

2026-07-13 기준으로 저장소에는 Spring Boot 애플리케이션 부트스트랩, `contextLoads()` 테스트 한 건, MySQL Docker Compose와 DB 연결 설정이 있다. 도메인, API, DB 마이그레이션과 Outbox 워커는 아직 구현되지 않았다. 따라서 아래 `UT-*`, `IT-*`, `AT-*`, `EXT-*`, `CT-*`, `PT-*` 항목은 별도로 **현재 증거**라고 표시하지 않는 한 모두 구현할 테스트 계획이다. 테스트 이름이나 문서만 존재하는 것은 통과 증거로 보지 않는다.

완료 기준은 한 문장으로 다음과 같다.

> 변경된 동작을 지우면 실패하는 테스트가 존재하고, 실제 MySQL 8.4 환경에서 관련 테스트·전체 테스트·검사·빌드·실행 확인이 모두 통과하며, 그 결과가 요구사항 추적표에 연결돼야 한다.

## 2. 검증 원칙

1. **불변식과 트랜잭션 경계를 먼저 검증한다.** 잔액 비음수, 중복 충전·주문 방지, 주문과 Outbox의 원자성이 우선순위다.
2. **운영 DB와 같은 제품을 사용한다.** SQL, 인덱스, 비관적 락, `SKIP LOCKED`, 제약 조건은 H2 같은 대체 DB가 아니라 MySQL 8.4에서 검증한다.
3. **현재 시각을 통제한다.** 기간 집계, JWT 만료, 멱등 레코드 보존, Outbox 재시도·임대·정리는 주입한 `Clock`과 명시적인 UTC 시각을 사용한다.
4. **비동기 동작을 시간 추측으로 검증하지 않는다.** 고정 `sleep` 뒤 결과를 기대하지 않고, 제한 시간 안에서 조건을 반복 확인하는 bounded polling을 사용한다.
5. **동시성 테스트는 실제로 분리된 트랜잭션에서 실행한다.** 같은 테스트 트랜잭션이나 같은 DB 커넥션을 공유한 결과를 동시성 증거로 인정하지 않는다.
6. **외부 경계는 계약까지 검증한다.** 외부 수집 API 자체는 Mock HTTP 서버로 대체하되, 실제 HTTP 직렬화, timeout, 상태 코드 분류, 중복 이벤트 ID를 확인한다.
7. **테스트는 독립적이고 진단 가능해야 한다.** 실행 순서, 로컬 시간대, 이전 테스트 데이터에 의존하지 않으며 실패 시 재현에 필요한 식별자와 관측값을 남긴다.
8. **정량 목표를 만들지 않는다.** 성능 목표치는 아직 `TBD`다. 우선 동일 조건에서 p50/p95/p99, 처리량, 락 대기, 실행 계획, Outbox backlog를 측정해 기준선을 만든다.

## 3. 테스트 ID와 상태 표기

테스트 ID는 요구사항, 테스트 코드, 실행 결과를 안정적으로 연결하는 식별자다.

| 접두사 | 계층 | 주된 목적 |
|---|---|---|
| `UT` | 단위 테스트 | DB·네트워크 없이 규칙과 분기 검증 |
| `IT` | 통합 테스트 | 실제 MySQL, Spring 구성, 마이그레이션 검증 |
| `AT` | API 인수 테스트 | HTTP 요청·응답·인증·오류 계약 검증 |
| `EXT` | 외부 연동 테스트 | Mock HTTP 서버와 Outbox 전달 정책 검증 |
| `CT` | 동시성 테스트 | 경합, 락, 다중 워커, fencing 검증 |
| `PT` | 성능 기준선 테스트 | 지연 분포, 처리량, DB·워커 병목 측정 |
| `QT` | 품질·구성 검사 | 비밀정보, 설정, 문서·스키마 일치 검증 |

ID 형식은 `<계층>-<영역>-<3자리 순번>`으로 고정한다. 예를 들어 `CT-ORDER-001`은 주문 동시성의 첫 번째 시나리오다.

- 이미 배정한 ID는 테스트를 삭제하더라도 다른 의미로 재사용하지 않는다.
- 테스트 메서드의 `@DisplayName` 또는 테스트 이름에 ID를 포함한다.
- 하나의 테스트가 여러 요구사항에 기여할 수 있지만, 한 ID 안에 서로 독립적인 성공·실패 조건을 과도하게 합치지 않는다.
- 추적표의 `계획` ID는 테스트 코드와 통과 결과가 모두 생긴 뒤에만 `증거`로 전환한다.
- 임시 비활성화, 조건부 제외, 재시도 후 우연한 성공은 통과로 숨기지 않고 사유와 건수를 보고한다.

## 4. 실행 환경과 계층

### 4.1 환경 계약

| 항목 | 계약 | 현재 상태 |
|---|---|---|
| Java | Amazon Corretto 17 호환 JDK | Gradle toolchain에 Java 17 지정 |
| 애플리케이션 | Spring Boot 4.1.0, Gradle 9.5.1 | 빌드 기준선 존재 |
| 데이터베이스 | Docker Compose의 MySQL 8.4 LTS | Compose와 접속 설정 구현, health check 확인 |
| 애플리케이션 실행 | 호스트에서 `./gradlew bootRun` | MySQL 연결과 애플리케이션 시작 확인 |
| 외부 수집 API | 테스트 프로세스가 제어하는 Mock HTTP 서버 | 미구현 |
| 저장 기준 시간 | UTC | 설정과 검증 미구현 |

Docker Compose는 MySQL만 실행한다. 애플리케이션과 Gradle 테스트는 호스트에서 실행해 디버깅과 반복 시간을 단순하게 유지한다. 새 테스트 라이브러리가 필요하면 프로젝트 경계 규칙에 따라 추가 전에 승인을 받는다.

### 4.2 계층별 책임

| 계층 | 포함 | 포함하지 않음 |
|---|---|---|
| 단위 | 금액 규칙, 해시 비교, JWT·시간 계산, 재시도 분류, 오류 매핑 | Spring context, DB, 실제 HTTP |
| DB 통합 | Flyway, JPA·SQL, 제약, 인덱스, 락, 트랜잭션 | 외부 API의 실제 가용성 |
| API 인수 | 필터·인증·검증·직렬화·HTTP 상태와 본문 | 외부 수집 시스템 구현 |
| 외부 연동 | 실제 HTTP 요청 형식, timeout, 재시도, 상태 전이 | 인터넷상의 실제 수집 서버 |
| 동시성 | 별도 커넥션·트랜잭션의 경합과 최종 DB 상태 | 단일 스레드 모의 객체 결과 |
| 성능 기준선 | 고정 데이터 규모에서 지연 분포·처리량·DB 지표 | 목표치가 없는 상태의 임의 합격/불합격 판정 |

### 5.4 외부 수집 API·Outbox 테스트

| ID | 시나리오 | 핵심 검증 |
|---|---|---|
| `EXT-OUTBOX-001` | 이벤트 요청 계약 | `eventId`, `ORDER_PAID`, 발생시각, 사용자·메뉴 ID, 결제금액 정확성, 인증 헤더 없음 |
| `EXT-OUTBOX-002` | 첫 전송 성공 | 커밋 후 즉시 대상이 되고 `2xx`에서 `PUBLISHED` 전환 |
| `EXT-OUTBOX-003` | 일시 실패 | 네트워크 오류·timeout·408·429·5xx에서 다음 시각과 시도 횟수 저장 |
| `EXT-OUTBOX-004` | 영구 HTTP 실패 | redirect 미추적, `3xx`와 408·429 제외 `4xx`에서 즉시 `FAILED` |
| `EXT-OUTBOX-005` | 최대 시도 소진 | 최초 포함 총 4회 실패 후 `FAILED`, 원본 주문 유지 |
| `EXT-OUTBOX-006` | 응답 유실과 중복 전송 | 재전송해도 동일 `eventId`를 사용해 수신 측 중복 제거 가능 |
| `EXT-OUTBOX-007` | 워커 중단 복구 | 30초 lease 만료 후 다른 워커가 같은 이벤트를 새 토큰으로 재처리 |

Mock 서버는 테스트마다 수신 기록과 응답 스크립트를 초기화한다. 성공 검증은 상태만 보지 않고 HTTP 메서드·경로·헤더·JSON 본문과 요청 횟수를 함께 확인한다.

### 5.5 동시성 테스트

| ID | 시나리오 | 핵심 검증 |
|---|---|---|
| `CT-POINT-001` | 같은 사용자 동시 충전 | 모든 성공 금액 합계와 최종 잔액 일치, 유실 갱신 없음 |
| `CT-POINT-002` | 서로 다른 사용자 동시 충전 | 사용자별 결과 독립, 불필요한 전역 직렬화 없음 |
| `CT-AUTH-001` | 같은 이메일 동시 회원가입 | 정확히 한 요청만 사용자 생성, 최종 사용자 행 1개, 나머지는 중복 오류 |
| `CT-APP-001` | 둘 이상의 애플리케이션 인스턴스 | 한 인스턴스가 발급한 토큰과 커밋한 상태를 다른 인스턴스가 동일하게 처리 |
| `CT-ORDER-001` | 5,000P에서 4,000P 주문 2건 | 정확히 한 건 성공, 잔액 1,000P, 주문·Outbox 각 1건 |
| `CT-IDEM-001` | 같은 키·같은 요청 동시 실행 | 업무 반영 1회, 두 호출의 확정 결과 동일 |
| `CT-IDEM-002` | 같은 키·다른 요청 동시 실행 | 한 요청만 선점, 다른 요청은 충돌, 중복 반영 없음 |
| `CT-OUTBOX-001` | 여러 워커의 같은 배치 선점 | 이벤트 하나당 활성 claim 1개, 누락·겹침 없음 |
| `CT-OUTBOX-002` | lease 경계의 늦은 워커 | 이전 `claim_token`으로 `PUBLISHED`·재시도 상태 덮어쓰기 불가 |

구현 절차는 다음과 같다.

1. 준비 데이터를 별도 트랜잭션으로 커밋한다.
2. 작업마다 별도 `EntityManager` 또는 DB 커넥션과 별도 트랜잭션을 연다.
3. `CountDownLatch` 같은 실행 장벽으로 시작 시점을 맞춘다.
4. 각 작업에 종료 제한 시간을 두고 성공값과 예외를 모두 수집한다.
5. 모든 작업이 끝난 뒤 새로운 조회 트랜잭션에서 잔액, 주문, 멱등 레코드, Outbox를 검증한다.
6. 실행기를 종료하고 테스트 데이터를 정리한다.

동시성 테스트 메서드 전체를 하나의 테스트 트랜잭션으로 감싸지 않는다. 그러면 준비 데이터가 작업 커넥션에 보이지 않거나 실제 커밋·락 경계가 왜곡된다.

### 5.6 성능 기준선 테스트

현재는 예상 동시 사용자, RPS, 운영 데이터 규모, 허용 지연이 확정되지 않았으므로 수치 합격선을 두지 않는다. 다음 결과를 동일한 형식으로 수집해 목표 수립의 입력으로 사용한다.

| ID | 부하 모양 | 반드시 기록할 값 |
|---|---|---|
| `PT-API-001` | 메뉴 조회·충전·주문 혼합 요청 | 요청별 p50/p95/p99, 처리량, 오류율, 측정 시간 |
| `PT-LOCK-001` | 같은 사용자 경합과 서로 다른 사용자 부하 비교 | 트랜잭션 지연, DB lock wait, timeout·deadlock 수 |
| `PT-POPULAR-001` | 최근 7일 주문을 포함한 여러 데이터 규모 | p50/p95/p99, rows examined, `EXPLAIN ANALYZE`, 선택 인덱스 |
| `PT-OUTBOX-001` | backlog가 있는 다중 워커 | backlog 크기, 초당 처리량, 생성→발행 지연, 재시도·FAILED 수 |

측정 전 데이터 생성과 Flyway 실행은 계측 구간에서 제외한다. 워밍업과 측정 구간을 분리하고, 반복 횟수·JDK·CPU·메모리·MySQL 설정·커넥션 풀·워커 수·데이터 규모를 결과와 함께 기록한다. 평균만 보고하지 않으며 성능 변화와 기능 오류를 구분한다. 목표치가 합의되기 전에는 측정값만으로 기능 완료를 임의 판정하지 않는다.

## 6. 시간·비동기 테스트 규칙

### 고정 Clock

- 테스트 기준 시각은 명시적인 UTC 값으로 고정한다. 예: `2026-07-11T12:00:00Z`.
- 인기 메뉴는 `기준-7일` 바로 전, 정확한 하한, 정확한 상한, 상한 직후 데이터를 각각 만든다.
- JWT는 만료 1초 전, 정확한 만료 시각, 만료 1초 후를 검증한다.
- 정리 작업을 호출할 때 멱등 레코드 24시간과 `PUBLISHED` 30일의 경계 직전·정확한 경계·경계 직후를 검증한다. 자동 실행 시각은 테스트 계약이 아니다.
- Outbox 다음 재시도와 30초 lease도 같은 Clock에서 계산한다.
- 쿼리가 DB의 `NOW()`에 숨게 의존하지 않도록 서비스가 계산한 `from`, `to`, 만료 기준을 명시적으로 전달한다.

### Bounded polling

고정 `Thread.sleep(...)`은 결과를 보장하지 못하고 테스트만 느리게 하므로 완료 조건으로 사용하지 않는다.

1. 단조 증가 시계 기준의 최종 deadline을 먼저 정한다.
2. 짧은 간격으로 DB 상태 또는 Mock 서버 수신 건수를 확인한다.
3. 조건이 충족되면 즉시 종료한다.
4. deadline을 넘기면 마지막 관측 상태와 관련 식별자를 포함해 실패한다.

재시도 시각 정책 자체는 고정 Clock으로 테스트하고 실제 1분·5분·30분을 기다리지 않는다. 스케줄러 통합 테스트에서는 due 이벤트를 직접 준비하거나 Clock을 전진시킨다.

## 7. 데이터 격리와 재현성

- 각 테스트 데이터는 테스트 ID와 실행별 고유 접미사를 포함한다. 이메일, 멱등키, 이벤트 ID가 다른 테스트와 충돌하지 않아야 한다.
- 단일 커넥션 DB 테스트는 롤백 격리를 사용할 수 있다. 독립 커넥션이 필요한 동시성·워커 테스트는 명시적으로 커밋하고 테스트 소유 데이터만 정리한다.
- FK 순서에 맞춘 정리 또는 테스트 전용 schema 재생성 중 하나를 일관되게 사용한다. 운영과 다른 DDL을 테스트 편의로 만들지 않는다.
- 테스트 순서와 병렬 실행 여부가 결과를 바꾸지 않아야 한다. 이전 테스트가 만든 ID나 잔액을 가정하지 않는다.
- Mock 서버, Clock, 실행기, HTTP 수신 기록은 테스트마다 초기화한다.
- 무작위 부하는 seed를 기록하고 실패 시 같은 seed로 재현할 수 있게 한다.

## 8. 실패 진단 정보

실패 메시지와 테스트 리포트에는 가능한 범위에서 다음 정보를 남긴다.

- 테스트 ID, 고정 기준 시각, 데이터 seed 또는 실행 접미사
- 사용자·주문·이벤트 ID, 멱등키의 안전한 축약값, `claim_token`의 안전한 축약값
- 각 작업의 시작·종료·성공·예외와 최종 DB 행 수·상태
- SQLState, deadlock 또는 lock wait timeout 여부, 실행한 쿼리의 파라미터 유형
- Mock 서버가 받은 요청 횟수·상태 코드·마지막 본문
- 성능 테스트의 환경, 데이터 규모, 워커·스레드·커넥션 수

비밀번호 원문, JWT 전체, 서명 키, DB 비밀번호는 어떤 실패 로그에도 남기지 않는다.

## 9. 실행 절차

표준 로컬 실행 흐름은 다음과 같다.

```bash
docker compose config
docker compose up -d
./gradlew test
./gradlew check
./gradlew build
./gradlew bootRun
```

- `docker compose ps` 또는 health check로 MySQL 준비 완료를 확인한 뒤 테스트를 시작한다.
- `./gradlew test` 결과는 전체 테스트 수, 성공, 실패, 제외 수를 그대로 보고한다.
- 현재 별도 Java lint 플러그인은 구성돼 있지 않다. 도입 전에는 `check`와 컴파일 결과만 보고하며 이를 “린트 통과”라고 과장하지 않는다.
- 현재 `bootRun`은 애플리케이션이 MySQL에 연결되어 시작되는지 확인한다. Flyway가 구현되면 migration 완료를, health endpoint가 구현되면 해당 응답까지 확인한다.
- 성능 기준선은 기능 테스트와 분리된 작업으로 실행해 일반 테스트의 안정성과 시간을 해치지 않는다.

## 10. 변경 완료 게이트

| 게이트 | 통과 조건 |
