# 테스트 전략

## 1. 문서 목적과 현재 범위

이 문서는 구현이 완료됐다고 주장하는 결과 보고서가 아니라, 구현 과정에서 요구사항 누락과 회귀를 막기 위한 **검증 계약**이다. 테스트 결과의 현재 상태와 요구사항별 증거는 [요구사항 추적표](./requirements-traceability.md)에서 관리한다.

2026-07-16 기준으로 저장소에는 Spring Boot·MySQL·Flyway 기반선, 메뉴·멱등 포인트 충전·주문·인기 메뉴 API, 주문당 Outbox 이벤트 원자 저장과 분산 전달 워커, 관측 경계가 구현돼 있다. 승인된 현재 범위의 필수 `UT-*`, `IT-*`, `AT-*`, `EXT-*`, `CT-*`, `QT-*`는 [요구사항 추적성](./requirements-traceability.md)에 실행 증거가 연결돼 있다. 예약된 정리 테스트와 합의되지 않은 `PT-*` 기준선은 현재 범위에서 제외하며, 테스트 이름이나 문서만 존재하는 것은 통과 증거로 보지 않는다.

완료 기준은 한 문장으로 다음과 같다.

> 변경된 동작을 지우면 실패하는 테스트가 존재하고, 실제 MySQL 8.4 환경에서 관련 테스트·전체 테스트·검사·빌드·실행 확인이 모두 통과하며, 그 결과가 요구사항 추적표에 연결돼야 한다.

## 2. 검증 원칙

1. **불변식과 트랜잭션 경계를 먼저 검증한다.** 잔액 비음수, 중복 충전·주문 방지, 주문과 Outbox의 원자성이 우선순위다.
2. **운영 DB와 같은 제품을 사용한다.** SQL, 인덱스, 비관적 락, `SKIP LOCKED`, 제약 조건은 H2 같은 대체 DB가 아니라 MySQL 8.4에서 검증한다.
3. **현재 시각을 통제한다.** 기간 집계, Outbox 재시도와 임대는 주입한 `Clock`과 명시적인 UTC 시각을 사용한다.
4. **비동기 동작을 시간 추측으로 검증하지 않는다.** 고정 `sleep` 뒤 결과를 기대하지 않고, 제한 시간 안에서 조건을 반복 확인하는 bounded polling을 사용한다.
5. **동시성 테스트는 실제로 분리된 트랜잭션에서 실행한다.** 같은 테스트 트랜잭션이나 같은 DB 커넥션을 공유한 결과를 동시성 증거로 인정하지 않는다.
6. **외부 경계는 계약까지 검증한다.** 외부 수집 API 자체는 Mock HTTP 서버로 대체하되, 실제 HTTP 직렬화, timeout, 상태 코드 분류, 중복 이벤트 ID를 확인한다.
7. **테스트는 독립적이고 진단 가능해야 한다.** 실행 순서, 로컬 시간대, 이전 테스트 데이터에 의존하지 않으며 실패 시 재현에 필요한 식별자와 관측값을 남긴다.
8. **승인되지 않은 정량 목표를 만들지 않는다.** 정상·backlog 없음 조건의 Outbox 최초 요청 2초 기준 외 성능 목표치는 `TBD`다. 우선 동일 조건에서 p50/p95/p99, 처리량, 락 대기, 실행 계획, Outbox backlog를 측정해 기준선을 만든다.

## 3. 테스트 ID와 상태 표기

테스트 ID는 요구사항, 테스트 코드, 실행 결과를 안정적으로 연결하는 식별자다.

| 접두사 | 계층 | 주된 목적 |
|---|---|---|
| `UT` | 단위 테스트 | DB·네트워크 없이 규칙과 분기 검증 |
| `IT` | 통합 테스트 | 실제 MySQL, Spring 구성, 마이그레이션 검증 |
| `AT` | API 인수 테스트 | HTTP 요청·응답·입력·오류 계약 검증 |
| `EXT` | 외부 연동 테스트 | Mock HTTP 서버와 Outbox 전달 정책 검증 |
| `CT` | 동시성 테스트 | 경합, 락, 다중 워커, fencing 검증 |
| `PT` | 성능 기준선 테스트 | 지연 분포, 처리량, DB·워커 병목 측정 |
| `QT` | 품질·구성 검사 | 비밀정보, 설정, 문서·스키마 일치 검증 |

ID 형식은 `<계층>-<영역>-<3자리 순번>`으로 고정한다. 예를 들어 `CT-ORDER-001`은 주문 동시성의 첫 번째 시나리오다.

- 이미 배정한 ID는 테스트를 삭제하더라도 다른 의미로 재사용하지 않는다.
- 테스트 메서드의 `@DisplayName` 또는 테스트 이름에 ID를 포함한다.
- 하나의 테스트가 여러 요구사항에 기여할 수 있지만, 한 ID 안에 서로 독립적인 성공·실패 조건을 과도하게 합치지 않는다.
- 추적표의 `계획` ID는 테스트 코드와 통과 결과가 모두 생긴 뒤에만 `증거`로 전환한다.
- 임시 비활성화, 조건부 제외, 재시도 후 우연한 성공은 통과로 숨기지 않고 사유와 건수를 보고한다.

## 4. 실행 환경과 계층

### 4.1 환경 계약

| 항목 | 계약 | 현재 상태 |
|---|---|---|
| Java | Amazon Corretto 17 호환 JDK | Gradle toolchain에 Java 17 지정 |
| 애플리케이션 | Spring Boot 4.1.0, Gradle 9.5.1 | 빌드 기준선 존재 |
| 데이터베이스 | Docker Compose의 MySQL 8.4 LTS, 개발·테스트 DB 분리 | 초기화 스크립트·테스트 프로필·Flyway migration 검증 완료 |
| 애플리케이션 실행 | 호스트에서 `./gradlew bootRun` | MySQL 연결과 애플리케이션 시작 확인 |
| 외부 수집 API | 테스트 프로세스가 제어하는 JDK Mock HTTP 서버 | 요청·timeout·상태 분류·재시도 검증 완료 |
| 저장 기준 시간 | UTC | 주입 가능 `Clock`, 애플리케이션·JDBC·DB 설정과 비 UTC JVM 저장 검증 완료 |

Docker Compose는 MySQL만 실행한다. 애플리케이션과 Gradle 테스트는 호스트에서 실행해 디버깅과 반복 시간을 단순하게 유지한다. 개발과 테스트는 하나의 Compose MySQL 인스턴스를 사용하되 개발용 `coffee_order_system`과 테스트용 `coffee_order_system_test`를 분리한다. 자동 구현은 멱등적인 `docker/mysql/init/01-create-test-database.sh`를 `/docker-entrypoint-initdb.d/01-create-test-database.sh`에 연결해 신규 볼륨에서 두 데이터베이스와 애플리케이션 사용자의 권한을 준비한다. 기존 볼륨에 테스트 데이터베이스가 없으면 `docker compose exec mysql bash /docker-entrypoint-initdb.d/01-create-test-database.sh`를 한 번 실행한다. 이 절차는 데이터베이스와 권한만 멱등적으로 준비하고 개발 데이터나 볼륨을 삭제하지 않는다. 테스트 프로필은 개발 URL을 상속하지 않고 테스트 JDBC URL을 명시한다. 각 테스트는 실행별 고유 식별자로 만든 자기 데이터만 FK 순서에 맞춰 정리하며 전체 `TRUNCATE`, schema 삭제와 `Flyway clean`을 사용하지 않는다. 빈 DB 마이그레이션은 CI나 새 Compose 볼륨에서 검증한다.

구현 단계에서 추가가 승인된 애플리케이션 의존성은 `spring-boot-starter-validation`, `spring-boot-starter-actuator`, Spring Boot 4의 Flyway 자동 구성 모듈인 `org.springframework.boot:spring-boot-flyway`, `org.flywaydb:flyway-core`, `org.flywaydb:flyway-mysql`이다. 빌드 도구로는 Spotless 플러그인과 Java 포매터가 승인됐다. Mock HTTP, bounded polling과 동시성 실행은 JDK 표준 기능으로 구현하며 그 밖의 라이브러리는 추가 전에 다시 승인받는다.

### 4.2 계층별 책임

| 계층 | 포함 | 포함하지 않음 |
|---|---|---|
| 단위 | 금액 규칙, 해시 비교, 시간 계산, 재시도 분류, 오류 매핑 | Spring context, DB, 실제 HTTP |
| DB 통합 | Flyway, JPA·SQL, 제약, 인덱스, 락, 트랜잭션 | 외부 API의 실제 가용성 |
| API 인수 | 입력 검증·직렬화·HTTP 상태와 본문 | 외부 수집 시스템 구현 |
| 외부 연동 | 실제 HTTP 요청 형식, timeout, 재시도, 상태 전이 | 인터넷상의 실제 수집 서버 |
| 동시성 | 별도 커넥션·트랜잭션의 경합과 최종 DB 상태 | 단일 스레드 모의 객체 결과 |
| 성능 기준선 | 고정 데이터 규모에서 지연 분포·처리량·DB 지표 | 목표치가 없는 상태의 임의 합격/불합격 판정 |

## 5. 계획된 테스트 카탈로그

### 5.1 단위 테스트

| ID | 시나리오 | 핵심 검증 |
|---|---|---|
| `UT-POINT-001` | 범위 안의 정수 포인트 충전 | `1원 = 1P`, 기존 잔액에 정확히 합산 |
| `UT-POINT-002` | 0·음수·signed `BIGINT` 범위 밖 충전과 잔액 덧셈 overflow | `INVALID_CHARGE_AMOUNT` 또는 `POINT_BALANCE_OVERFLOW`, 잔액 상태 불변 |
| `UT-POINT-003` | 충분한 잔액에서 결제 | 결제금액만 차감하고 잔액 비음수 유지 |
| `UT-POINT-004` | 잔액 부족 결제 | `INSUFFICIENT_POINT`, 잔액 상태 불변 |
| `UT-IDEM-001` | 요청 해시 결정성 | `CHARGE|<amount>`, `ORDER|<menuId>` UTF-8 SHA-256 결과가 결정적임 |
| `UT-IDEM-002` | 같은 키의 다른 요청 내용 | 다른 메뉴·금액은 다른 해시로 식별 |
| `UT-OUTBOX-001` | 시도 순서별 재시도 시각과 실패 횟수 | 즉시 첫 시도 후 실패 시 1분·5분·30분 간격, 현재 claim의 영구 실패 포함 모든 반영 실패에서 count 증가, 총 4회 시도 |
| `UT-OUTBOX-002` | 외부 응답 분류 | `2xx` 성공, 네트워크 오류·timeout·`408`·`429`·`5xx` 재시도, `3xx`·그 외 `4xx` 즉시 실패 |
| `UT-OUTBOX-003` | timeout 구성 | 연결 2초, 요청 전체 5초와 redirect 금지가 JDK HTTP 클라이언트 설정에 반영 |
| `UT-POPULAR-001` | 인기 메뉴 구간 계산 | 고정 Clock 기준 `[현재-7일, 현재)` 계산 |
| `UT-POPULAR-002` | 인기 메뉴 순위 | 주문 수 내림차순, 동률 시 메뉴 ID 오름차순, 최대 3개 |
| `UT-API-001` | 공통 성공·오류 매핑 | 도메인 오류와 HTTP 상태·응답 코드의 일관성 |

| `IT-ORDER-003` | Outbox 또는 DB 인프라 오류 | 업무 변경과 멱등 레코드를 포함한 전체 트랜잭션 롤백 |
| `IT-IDEM-001` | 같은 키·같은 요청 재실행 | 저장된 최초 성공 또는 예상 가능한 실패 응답을 그대로 반환 |

| `IT-POPULAR-001` | 7일·상태 경계 집계 | 하한 포함, 상한 제외, `PAID` 주문만 집계 |

| `IT-POPULAR-003` | 메뉴 메타데이터 변경 | 주문 수는 주문 원본에서 집계하되 이름·가격은 현재 `menus` 값 반환 |

| `IT-OUTBOX-002` | 30초 임대 만료와 fencing | 만료 이벤트는 재선점되고 이전 토큰 워커의 늦은 상태 갱신은 거절 |
| `IT-OUTBOX-003` | Outbox 보존 정책 | 예약 ID — 정리 기능을 도입할 때 구현하며 현재 완료 게이트에서 제외 |
| `IT-RESILIENCE-001` | DB 경합 timeout·deadlock | 전체 롤백, 멱등 결과 없음, `TEMPORARILY_UNAVAILABLE` 매핑 |
| `IT-TIME-001` | 애플리케이션·DB 시간 기준 | 저장값과 기간 파라미터가 UTC이며 JVM 기본 시간대에 영향받지 않음 |

### 5.3 API 인수 테스트

| ID | 시나리오 | 핵심 검증 |
|---|---|---|
| `AT-MENU-001` | 메뉴 목록 | `menuId`, 현재 이름·가격을 ID 오름차순으로 반환 |
| `AT-USER-001` | 사용자 ID 계약 | 충전·주문 본문의 양의 정수 `userId`를 사용하고 사용자 부재는 `404 USER_NOT_FOUND`, 업무·멱등 상태 불변 |
| `AT-POINT-001` | 포인트 충전 성공 | 요청한 기존 사용자만 변경하고 충전액·최종 잔액 반환 |

구현 절차는 다음과 같다.

1. 준비 데이터를 별도 트랜잭션으로 커밋한다.
2. 작업마다 별도 `EntityManager` 또는 DB 커넥션과 별도 트랜잭션을 연다.
3. `CountDownLatch` 같은 실행 장벽으로 시작 시점을 맞춘다.
4. 각 작업에 종료 제한 시간을 두고 성공값과 예외를 모두 수집한다.
5. 모든 작업이 끝난 뒤 새로운 조회 트랜잭션에서 잔액, 주문, 멱등 레코드, Outbox를 검증한다.
6. 실행기를 종료하고 테스트 데이터를 정리한다.

동시성 테스트 메서드 전체를 하나의 테스트 트랜잭션으로 감싸지 않는다. 그러면 준비 데이터가 작업 커넥션에 보이지 않거나 실제 커밋·락 경계가 왜곡된다.

## 6. 시간·비동기 테스트 규칙

### 고정 Clock

- 테스트 기준 시각은 명시적인 UTC 값으로 고정한다. 예: `2026-07-11T12:00:00Z`.
- 인기 메뉴는 `기준-7일` 바로 전, 정확한 하한, 정확한 상한, 상한 직후 데이터를 각각 만든다.
- 멱등 레코드와 `PUBLISHED` 이벤트 정리 테스트는 실제 정리 기능을 도입할 때 추가한다. 현재 애플리케이션은 두 데이터를 삭제하지 않는다.
- Outbox 다음 재시도와 30초 lease도 같은 Clock에서 계산한다.
- 쿼리가 DB의 `NOW()`에 숨게 의존하지 않도록 서비스가 계산한 `from`, `to`, 만료 기준을 명시적으로 전달한다.

### Bounded polling

고정 `Thread.sleep(...)`은 결과를 보장하지 못하고 테스트만 느리게 하므로 완료 조건으로 사용하지 않는다. Outbox 2초 최초 요청 계약도 주문 커밋 시각을 기록한 뒤 Mock API 수신 시각을 제한 시간 안에서 관찰해 검증한다.

1. 단조 증가 시계 기준의 최종 deadline을 먼저 정한다.
2. 짧은 간격으로 DB 상태 또는 Mock 서버 수신 건수를 확인한다.
3. 조건이 충족되면 즉시 종료한다.
4. deadline을 넘기면 마지막 관측 상태와 관련 식별자를 포함해 실패한다.

재시도 시각 정책 자체는 고정 Clock으로 테스트하고 실제 1분·5분·30분을 기다리지 않는다. 스케줄러 통합 테스트에서는 due 이벤트를 직접 준비하거나 Clock을 전진시킨다.

## 7. 데이터 격리와 재현성

- 각 테스트 데이터는 테스트 ID와 실행별 고유 접미사를 포함한다. 멱등키와 이벤트 ID가 다른 테스트와 충돌하지 않아야 한다.
- 모든 통합 테스트는 `coffee_order_system_test`에만 접속하며 개발 데이터베이스 URL을 상속하거나 대체 경로로 사용하지 않는다.
- 단일 커넥션 DB 테스트는 롤백 격리를 사용할 수 있다. 독립 커넥션이 필요한 동시성·워커 테스트는 명시적으로 커밋하고 테스트 소유 데이터만 정리한다.
- 일반 테스트는 Outbox 워커를 비활성화한다. Outbox 전용 테스트만 테스트 프로세스의 Mock 서버를 준비한 뒤 워커를 활성화하고 종료 시 워커와 수신 기록을 함께 정리한다.
- FK 순서에 맞춰 해당 테스트가 만든 행만 정리한다. 테스트 데이터베이스 전체를 `TRUNCATE`하거나 schema를 삭제하거나 `Flyway clean`을 실행하지 않는다.
- 테스트 순서와 병렬 실행 여부가 결과를 바꾸지 않아야 한다. 이전 테스트가 만든 ID나 잔액을 가정하지 않는다.
- Mock 서버, Clock, 실행기, HTTP 수신 기록은 테스트마다 초기화한다.
- 무작위 부하는 seed를 기록하고 실패 시 같은 seed로 재현할 수 있게 한다.

## 9. 실행 절차

표준 로컬 실행 흐름은 다음과 같다.

```bash
cp .env.example .env
# .env의 비밀번호와 비밀값을 실행 환경에 맞게 변경
docker compose config
docker compose up -d --wait
# 기존 볼륨에 테스트 DB가 없는 경우에만 다음 명령을 한 번 실행
docker compose exec mysql bash /docker-entrypoint-initdb.d/01-create-test-database.sh
./gradlew test
./gradlew check
./gradlew build
./gradlew bootRun
```

로컬 실행에서는 `.env.example`을 `.env`로 복사해 필요한 비밀값을 설정한다. CI·운영에서는 파일 대신 같은 이름의 환경 변수를 주입할 수 있다.

- `docker compose up -d --wait`가 MySQL health check를 통과한 뒤 테스트를 시작한다.
- 신규 볼륨은 테스트 DB를 자동 생성한다. 기존 볼륨용 초기화 명령은 멱등적이며 실행해도 개발 데이터와 볼륨을 삭제하지 않는다.
- `./gradlew test` 결과는 전체 테스트 수, 성공, 실패, 제외 수를 그대로 보고한다.
- Spotless 구현 후 `./gradlew check`가 `spotlessCheck`와 테스트를 함께 실행한다. `spotlessApply`는 명시적으로만 실행하고 적용 뒤 전체 diff를 검토한다.
- `bootRun`은 애플리케이션의 MySQL 연결, Flyway migration 완료와 `/actuator/health`의 `UP` 응답까지 확인한다.
- 성능 기준선은 기능 테스트와 분리된 작업으로 실행해 일반 테스트의 안정성과 시간을 해치지 않는다.

## 10. 변경 완료 게이트

| 게이트 | 통과 조건 |
|---|---|
| 요구사항 | 정상 경로 1개와 실패·경계 경로 1개 이상이 대응 테스트 ID로 연결됨 |
| 회귀 방지 | 변경 구현을 제거하거나 조건을 뒤집으면 대응 테스트가 실패함 |
| 데이터 | MySQL 8.4에서 migration, 제약, 트랜잭션, 락 테스트 통과 |
| 동시성 | 별도 커넥션·트랜잭션에서 최종 불변식과 작업 건수 검증 |
| 외부 연동 | 요청 계약, 정상 조건의 2초 최초 요청, 분류, 실패 횟수, 상태별 필드 정리, 재시도, lease, fencing, 보존 정책 검증 |
| 시간 | 고정 Clock으로 모든 포함·제외 경계 검증, 로컬 시간대 의존 없음 |
| 품질 | 실패·제외 0, 불안정 재실행으로 결과를 숨기지 않음 |
| 검사·빌드 | Spotless 포맷 검사를 포함한 `check`와 `build` 통과 |
| 실행 | Docker MySQL + 호스트 `bootRun`으로 시작과 기본 상태 확인 |
| 추적성 | [요구사항 추적표](./requirements-traceability.md)의 구현 상태와 실행 증거 갱신 |

한 게이트라도 확인하지 못했으면 “완료”가 아니라 미검증 항목과 이유를 명시한다.

`PT-*` 부하·성능 기준선은 이 첫 완료 게이트와 분리한다. 다만 정상·backlog 없음 조건의 Outbox 최초 HTTP 요청 시작 2초는 기능 계약이므로 `EXT-OUTBOX-008`로 첫 완료 게이트에서 검증한다.
