# Product Requirements Document

## 1. 문서 목적과 상태

포인트로 커피를 주문하고 최근 인기 메뉴를 조회하는 REST API 백엔드의 목표 동작과 완료 기준을 정의합니다. 여러 애플리케이션 인스턴스가 같은 MySQL을 사용할 때도 포인트·주문·외부 이벤트의 정합성을 유지해야 합니다.

> 이 문서는 **목표 제품 계약**입니다. 현재 저장소는 Spring Boot 부트스트랩 단계이며 아래 기능은 아직 구현 완료 상태가 아닙니다. 요구사항이 구현됐다는 판정은 대응 테스트, 정적 검사, 빌드와 실행 확인이 모두 끝난 뒤에만 내립니다.

용어는 [도메인 컨텍스트](./context.md), HTTP 필드는 [API 명세](./api-spec.md), 데이터 구조는 [ERD](./erd.md), 설계 결정은 [ADR 목록](./adr/)을 기준으로 합니다.

## 2. 제품 목표와 성공 조건

| 목표 | 성공 조건 |
|---|---|
| 안전한 포인트 변경 | 동시 충전·결제 후에도 성공 금액 합계와 최종 잔액이 일치하고 잔액이 음수가 되지 않음 |
| 중복 변경 방지 | 네트워크 재시도로 같은 충전·주문이 들어와도 하나의 논리 요청이 한 번만 반영됨 |
| 원자적 주문 | 포인트 차감, 결제 완료 주문, Outbox 이벤트가 모두 함께 커밋되거나 모두 롤백됨 |
| 외부 장애 격리 | 외부 데이터 수집 API의 지연·오류가 이미 확정 가능한 고객 주문의 성공을 막지 않음 |
| 정확한 인기 메뉴 | 최근 7×24시간 `PAID` 주문을 기준으로 상위 3개와 결정적인 동률 순서를 반환함 |

Outbox의 정상 조건 최초 HTTP 요청 시작은 주문 커밋 후 2초 이내로 확정했습니다. 그 밖의 정량 성능 목표는 아직 확정하지 않았으며, 먼저 이 문서의 측정 시나리오로 기준값을 수집하고 사용자·트래픽·데이터 규모 가정과 함께 별도로 승인받습니다.

## 3. 사용자와 핵심 시나리오

### 사용자

- 이메일과 비밀번호로 가입합니다.
- 로그인해 30분 동안 유효한 JWT Access Token을 발급받습니다.
- 인증된 사용자 본인의 포인트만 충전하고 사용할 수 있습니다.

### 정상 주문 시나리오

1. 사용자가 가입하고 로그인합니다.
2. 사용자가 현재 메뉴 목록을 조회합니다.
3. 사용자가 고유한 `Idempotency-Key`로 양의 정수 포인트를 충전합니다.
4. 사용자가 새로운 `Idempotency-Key`와 메뉴 ID 하나로 주문합니다.
5. 시스템이 사용자 행을 잠그고 현재 메뉴 가격과 포인트 잔액을 확인합니다.
6. 시스템이 잔액을 차감하고 주문 스냅샷과 Outbox 이벤트를 같은 트랜잭션에 저장합니다.
7. 주문 응답 후 Outbox 워커가 외부 데이터 수집 API로 주문 이벤트를 전달합니다.
8. 사용자가 최근 7일 인기 메뉴 3개를 조회합니다.

## 4. 공통 정책

### 4.1 인증과 소유권

- 보호 API는 `Authorization: Bearer <access-token>`에서 인증 사용자 ID를 얻습니다.
- 요청 본문으로 사용자 ID를 받거나 신뢰하지 않습니다.
- 사용자는 자신의 포인트만 충전하고 자신의 잔액으로만 주문합니다.
- 원문 요구사항의 "사용자 식별값"은 검증된 JWT의 `sub`에 대응합니다.

### 4.2 금액과 시간

- 포인트와 모든 금액은 정수이며 1원은 1P입니다.
- 애플리케이션과 MySQL의 저장 기준 시간은 UTC입니다.
- 외부로 반환하는 날짜·시간은 ISO 8601 UTC 형식입니다.
- 인기 메뉴의 최근 7일은 달력 날짜가 아닌 정확한 7×24시간입니다.

### 4.3 동시성과 트랜잭션

- 충전과 주문은 사용자 PK의 DB 비관적 락을 획득한 뒤 잔액을 변경합니다.
- 락 범위는 사용자 한 행이며 외부 HTTP 호출을 DB 트랜잭션에 포함하지 않습니다.
- 애플리케이션 로컬 락이나 서버 로컬 세션은 정합성 수단으로 사용하지 않습니다.

### 4.4 멱등성

- 포인트 충전과 주문에는 `Idempotency-Key`가 필수입니다.
- 멱등 범위는 인증 사용자·작업 유형·멱등키 조합입니다.
- 같은 키와 같은 요청은 최초 확정 HTTP 상태와 응답 본문을 재사용합니다.
- 같은 키에 다른 요청 내용을 보내면 `409 IDEMPOTENCY_KEY_REUSED`로 거절합니다.
- 비즈니스 변경과 멱등 레코드는 하나의 DB 트랜잭션으로 처리합니다.
- 성공과 예상 가능한 비즈니스 실패는 결과를 `COMPLETED`로 저장하고 커밋합니다.
- 예상하지 못한 인프라 오류는 비즈니스 변경과 멱등 레코드를 모두 롤백합니다.
- `Idempotency-Key`는 UUID 문자열 형식이어야 합니다.
- 요청 지문은 작업 유형과 검증된 `amount` 또는 `menuId`를 조합한 SHA-256입니다.
- 인증·헤더·JSON·필드·충전 금액 검증 실패는 멱등 결과로 저장하지 않습니다.
- `COMPLETED` 레코드는 `completed_at`부터 최소 24시간 보존하고 이후 삭제할 수 있습니다. 자동 정리 시각은 보장하지 않으며 실제로 삭제된 키만 새 요청으로 처리될 수 있습니다.

상태 전이와 재응답 범위는 [API 명세의 멱등성 계약](./api-spec.md#10-멱등성-계약)을 따릅니다.

## 5. 기능 요구사항

### FR-01 회원가입

**목적**

새 사용자가 이메일과 비밀번호로 인증 계정을 만들 수 있어야 합니다.

**입력과 선행 조건**

- 입력: `email`, `password`
- 두 필드는 요청에 필요합니다. 이메일은 앞뒤 공백 제거 후 `@NotBlank`, `@Email`과 최대 254자로 검증하고, 저장한 대소문자는 유지하되 중복 비교는 대소문자를 무시합니다.
- 비밀번호는 최소 8자, UTF-8 기준 최대 72바이트입니다. 조합 규칙은 두지 않으며 앞뒤 공백이 있으면 거절하고 중간 공백은 허용합니다.
- 같은 이메일의 사용자가 없어야 합니다. 최종 중복 방지는 DB 유일 제약 조건이 담당합니다.

**성공 후조건**

- 사용자 한 명이 생성되고 `201 USER_CREATED`를 반환합니다.
- 신규 사용자의 포인트 잔액은 `0P`입니다.
- 비밀번호는 BCrypt 단방향 해시만 저장합니다.
- 비밀번호 원문이나 해시는 API 응답에 포함하지 않습니다.

**실패 후조건**

- 이메일이 이미 존재하면 `409 EMAIL_ALREADY_EXISTS`를 반환하고 새 사용자를 만들지 않습니다.
- 형식 또는 필드 검증에 실패하면 `400 INVALID_REQUEST`를 반환하고 사용자를 만들지 않습니다.
- 비밀번호 원문은 실패 로그에도 남기지 않습니다.

**수용 기준**

1. 사용하지 않은 이메일로 요청하면 사용자 ID와 이메일을 반환합니다.
2. 같은 이메일로 순차 또는 동시 요청해도 최종 사용자 행은 하나뿐입니다.
3. 저장소와 애플리케이션 로그 어디에도 비밀번호 원문이 존재하지 않습니다.
4. BCrypt 해시로 올바른 비밀번호는 검증되고 다른 비밀번호는 거절됩니다.

### FR-02 로그인

**목적**

가입 사용자가 보호 API에 사용할 수 있는 짧은 수명의 인증 토큰을 얻어야 합니다.

**입력과 선행 조건**

- 입력: `email`, `password`
- 사용자와 BCrypt 비밀번호 해시가 존재해야 합니다.

**성공 후조건**

- 올바른 자격 증명에 `200 LOGIN_SUCCESS`를 반환합니다.
- JWT Access Token은 발급 후 1,800초 동안 유효합니다.
- 보호 API는 토큰에서 복원한 사용자 ID를 인증 주체로 사용합니다.

**실패 후조건**

- 이메일 부재와 비밀번호 불일치는 모두 `401 INVALID_CREDENTIALS`로 응답합니다.
- 실패 응답으로 이메일 존재 여부를 구분할 수 없게 합니다.

**수용 기준**

1. 올바른 자격 증명은 `Bearer` 타입 토큰과 `expiresInSeconds: 1800`을 반환합니다.
2. 잘못된 이메일과 잘못된 비밀번호는 같은 공개 오류 코드와 메시지 정책을 사용합니다.
3. 누락·만료·위조 토큰으로 보호 API를 호출하면 `401 AUTHENTICATION_REQUIRED`를 반환합니다.
4. 요청 또는 인증 로그에 비밀번호와 JWT 원문을 남기지 않습니다.

### FR-03 메뉴 목록 조회

**목적**

클라이언트가 주문 가능한 현재 메뉴와 현재 가격을 확인할 수 있어야 합니다.

**입력과 선행 조건**

- 인증이나 요청 본문 없이 `GET /api/v1/menus`를 호출합니다.
- 초기 메뉴 데이터는 Flyway 마이그레이션으로 구성합니다.

| ID | 메뉴명 | 가격 |
|---:|---|---:|
| 1 | 아메리카노 | 4,000P |
| 2 | 카페라떼 | 5,000P |
| 3 | 카푸치노 | 5,500P |

**성공 후조건**

- 각 메뉴의 ID, 현재 이름, 현재 가격을 반환합니다.
- 전체 결과는 메뉴 ID 오름차순입니다.
- 조회는 사용자·주문·포인트 상태를 변경하지 않습니다.

**실패 후조건**

- 예상하지 못한 인프라 오류 외에는 도메인 상태를 변경하지 않습니다.

**수용 기준**

1. 응답은 `200 MENUS_RETRIEVED`와 메뉴 배열을 반환합니다.
2. 데이터 입력 순서와 관계없이 메뉴 ID가 오름차순입니다.
3. 가격은 정수 포인트이며 DB의 현재 `menus.price`와 일치합니다.
4. 인증 헤더 없이 조회할 수 있습니다.

### FR-04 포인트 충전

**목적**

인증 사용자가 중복 반영 위험 없이 자신의 현재 포인트 잔액을 늘릴 수 있어야 합니다.

**입력과 선행 조건**

- 인증: 유효한 JWT Access Token
- 헤더: `Idempotency-Key`
- 입력: signed `BIGINT` 범위의 0보다 큰 정수 `amount`

**성공 후조건**

- 사용자 행을 비관적으로 잠근 상태에서 기존 잔액에 `amount`를 더합니다.
- 잔액 변경과 `200 POINT_CHARGED` 응답 결과를 같은 트랜잭션으로 커밋합니다.
- 응답의 `chargedAmount`는 요청 금액, `balance`는 커밋된 최종 잔액입니다.

**실패 후조건**

- 인증 실패는 `401 AUTHENTICATION_REQUIRED`이며 잔액을 변경하지 않습니다.
- `amount`가 정수가 아니거나 0 이하이거나 signed `BIGINT` 범위를 벗어나면 `400 INVALID_CHARGE_AMOUNT`이며 잔액을 변경하지 않습니다.
- 현재 잔액과 유효한 `amount`의 합이 signed `BIGINT` 범위를 벗어나면 `409 POINT_BALANCE_OVERFLOW`이며 잔액을 변경하지 않고 실패 결과를 멱등 저장합니다.
- 같은 키에 다른 요청 내용을 사용하면 `409 IDEMPOTENCY_KEY_REUSED`이며 잔액을 변경하지 않습니다.
- DB 락 대기가 5초를 넘거나 deadlock이 발생하면 전체를 롤백하고 `503 TEMPORARILY_UNAVAILABLE`을 반환합니다. 이 결과는 멱등 저장하지 않습니다.
- 인프라 오류는 잔액 변경과 멱등 레코드를 모두 롤백합니다.

**수용 기준**

1. 10,000P 충전 성공 후 잔액이 정확히 10,000P 증가합니다.
2. 같은 키와 같은 금액을 반복 또는 동시에 요청해도 잔액은 한 번만 증가하고 최초 응답을 재사용합니다.
3. 같은 키와 다른 금액을 요청하면 추가 증가 없이 409를 반환합니다.
4. 같은 사용자에 대한 여러 충전이 동시에 성공하면 최종 잔액 증가분은 성공 금액의 합과 같습니다.
5. 서로 다른 사용자의 충전은 한 사용자 행 락을 공유하지 않습니다.
6. 임의의 업무 상한은 두지 않으며 signed `BIGINT` 경계와 덧셈 overflow를 상태 변경 없이 거절합니다.

### FR-05 주문 및 결제

**목적**

인증 사용자가 현재 메뉴 하나를 포인트로 결제하고, 잔액·주문·외부 전달 사건을 원자적으로 확정해야 합니다.

**입력과 선행 조건**

- 인증: 유효한 JWT Access Token
- 헤더: `Idempotency-Key`
- 입력: 현재 존재하는 메뉴의 `menuId` 하나
- 사용자 잔액은 주문 시점의 현재 메뉴 가격 이상이어야 합니다.

**성공 후조건**

- 결제금액은 주문 처리 시 조회한 현재 `menus.price`입니다.
- 사용자 행 락 안에서 결제금액을 차감하고 잔액이 음수가 아님을 확인합니다.
- 주문에 메뉴 ID, 주문 시점 메뉴명, 결제금액과 `PAID` 상태를 저장합니다.
- 주문 하나에 `ORDER_PAID` Outbox 이벤트 하나를 저장합니다.
- 포인트 차감, 주문, Outbox, `201 ORDER_PAID` 멱등 결과를 하나의 트랜잭션으로 커밋합니다.

**실패 후조건**

- 메뉴가 없으면 `404 MENU_NOT_FOUND`, 잔액이 부족하면 `409 INSUFFICIENT_POINT`를 반환합니다.
- 위 예상 가능한 실패는 포인트·주문·Outbox를 변경하지 않고 실패 결과만 멱등 레코드에 저장해 커밋합니다.
- 같은 멱등키에 다른 메뉴를 사용하면 `409 IDEMPOTENCY_KEY_REUSED`이며 추가 결제하지 않습니다.
- DB·커밋·Outbox 저장 등 인프라 오류는 포인트·주문·Outbox·멱등 레코드를 모두 롤백합니다.
- DB 락 대기가 5초를 넘거나 deadlock이 발생하면 전체를 롤백하고 `503 TEMPORARILY_UNAVAILABLE`을 반환합니다.

**수용 기준**

1. 잔액 5,000P로 4,000P 메뉴를 주문하면 잔액 1,000P, `PAID` 주문 1건, Outbox 1건이 남습니다.
2. 잔액 부족 시 잔액이 그대로이며 주문과 Outbox가 생성되지 않습니다.
3. 잔액 5,000P에서 4,000P 주문 두 건을 동시에 요청하면 한 건만 성공하고 잔액은 1,000P입니다.
4. 동일 키·동일 메뉴 재요청은 추가 차감 없이 최초 HTTP 상태와 응답 본문을 반환합니다.
5. 주문 이후 현재 메뉴가 바뀌어도 `menu_name_snapshot`과 `paid_amount`는 바뀌지 않습니다.
6. Outbox 저장이 실패하면 포인트 차감과 주문도 커밋되지 않습니다.

### FR-06 주문 데이터 전송

**목적**

고객 주문의 성공을 외부 시스템 가용성과 분리하면서 결제 완료 사건을 유실하지 않고 전달해야 합니다.

**입력과 선행 조건**

- 주문과 함께 커밋된 `PENDING` Outbox 이벤트가 있어야 합니다.
- 외부 요청에는 이벤트 ID, 이벤트 유형, 발생 시각, 사용자 ID, 메뉴 ID, 결제금액을 포함합니다.
- base URL은 기본값 없는 `COLLECTION_API_BASE_URL` 환경 변수로 주입하고 누락되거나 잘못되면 애플리케이션 시작이 실패합니다. 현재 범위에서는 인증 헤더를 사용하지 않습니다.

**성공 후조건**

- 워커는 주문 커밋과 동시에 전송 가능한 이벤트를 만들고, 워커 정상·전송 가능한 기존 backlog 없음 조건에서 2초 이내에 첫 외부 HTTP 요청을 시작합니다.
- 2초 기준은 외부 응답 완료나 `PUBLISHED` 전환이 아니라 요청 시작 시각에 적용합니다.
- 외부 응답이 `2xx`이면 이벤트를 `PUBLISHED`로 전환합니다.
- `PUBLISHED` 이벤트는 발행 완료 후 최소 30일 보존한 뒤 삭제할 수 있습니다. 자동 정리 스케줄러는 현재 범위에 포함하지 않습니다.

**실패와 복구 후조건**

- 네트워크·I/O 오류, timeout, `408`, `429`, `5xx`는 재시도 대상입니다.
- 나머지 `4xx`는 재시도하지 않고 즉시 `FAILED`로 전환합니다.
- redirect는 따르지 않고 `3xx`도 재시도 없이 즉시 `FAILED`로 전환합니다.
- 재시도 대상은 즉시 첫 시도 후 1분, 5분, 30분 간격으로 총 4회 시도합니다.
- 네 번째 시도도 재시도 가능한 오류로 실패하면 `FAILED`로 전환합니다.
- 연결 timeout은 2초, 요청 전체 timeout은 5초, 이벤트 lease는 30초입니다.
- `FAILED` 이벤트에는 `failed_at`, 선택적인 `last_http_status`, `last_error_type`만 저장하고 응답 본문·stack trace는 저장하지 않습니다.
- `FAILED` 이벤트는 자동 삭제하지 않으며 수동 재시도 API는 현재 범위에 포함하지 않습니다.
- 외부 전송 실패는 이미 커밋된 주문과 포인트를 되돌리지 않습니다.

**수용 기준**

1. 주문 API는 외부 API 응답을 기다리지 않고 주문 트랜잭션 결과를 반환합니다.
2. 여러 워커가 실행돼도 `SKIP LOCKED`로 서로 잠근 이벤트를 건너뜁니다.
3. 완료 갱신은 현재 `claim_token`이 일치할 때만 성공해 만료된 워커가 새 결과를 덮어쓰지 못합니다.
4. 각 오류 분류에 따라 정확히 재시도 또는 즉시 `FAILED` 전환합니다.
5. 전송 요청의 `eventId`는 재시도 사이에도 같아 수신 측이 중복 제거할 수 있습니다.
6. 워커가 중단되면 30초 lease 만료 후 다른 워커가 이벤트를 다시 선점할 수 있습니다.
7. 워커는 활성 배치가 없을 때 기본 1초마다 `next_retry_at`, `created_at`, `event_id` 순으로 최대 50건을 선점하고 비동기로 병렬 전송합니다.
8. 정상·backlog 없음 조건의 첫 HTTP 요청은 주문 커밋 후 2초 이내 시작하며 주문 API는 외부 응답을 기다리지 않습니다.

### FR-07 인기 메뉴 조회

**목적**

클라이언트가 조회 시점 기준의 정확한 최근 인기 메뉴 상위 3개를 확인할 수 있어야 합니다.

**입력과 선행 조건**

- 인증이나 요청 본문 없이 `GET /api/v1/menus/popular`를 호출합니다.
- 한 번 고정한 조회 시각을 상한 `to`, `to - 7일`을 하한 `from`으로 사용합니다.

**성공 후조건**

- `[from, to)` 구간의 `PAID` 주문만 메뉴 ID별로 집계합니다.
- 주문 수 내림차순, 동률이면 메뉴 ID 오름차순으로 정렬해 최대 3개를 반환합니다.
- `orderCount`는 결제 완료 주문에서 계산합니다.
- 응답의 `name`, `price`는 과거 주문 스냅샷이 아니라 조회 시점의 현재 `menus`에서 가져옵니다.
- 대상이 3개 미만이면 존재하는 항목만, 대상이 없으면 빈 배열을 반환합니다.

**실패 후조건**

- 조회는 포인트·주문·메뉴·Outbox 데이터를 변경하지 않습니다.

**수용 기준**

1. 정확히 하한 시각에 결제된 주문은 포함하고 상한 시각의 주문은 제외합니다.
2. 취소 상태를 포함한 미래 상태가 생겨도 `PAID`가 아닌 주문은 세지 않습니다.
3. 주문 수가 같은 메뉴는 메뉴 ID 오름차순으로 순위가 결정됩니다.
4. 주문 당시 메뉴명·가격과 현재 메뉴가 다르면 현재 메뉴명·가격을 표시하되 주문 수는 그대로입니다.
5. 데이터 입력 순서나 반복 조회에 관계없이 같은 기준 시각·데이터에는 같은 순서를 반환합니다.

## 6. 비기능 요구사항

### NFR-01 데이터 일관성

- 공용 MySQL을 포인트·주문·멱등 결과·전달 상태의 단일 진실 원천으로 사용합니다.
- DB 유일 제약과 CHECK 제약으로 이메일·멱등키 중복과 음수 잔액을 방어합니다.
- 실제 MySQL 통합 테스트로 비관적 락, `SKIP LOCKED`, 트랜잭션, 인덱스와 제약 조건을 검증합니다.
- Outbox 외부 호출은 고객 요청 트랜잭션 밖에서 실행합니다.

### NFR-02 성능과 용량 측정

워커 정상·전송 가능한 기존 backlog 없음 조건의 Outbox 최초 HTTP 요청 시작은 주문 커밋 후 2초 이내입니다. 그 밖의 정량 목표는 **미정(TBD)**이며 근거 없는 응답시간이나 처리량 수치를 요구사항으로 만들지 않습니다. 첫 구현 후 아래 항목을 동일한 환경·데이터·부하 조건에서 측정하고 원시 결과와 함께 기록합니다.

| 관점 | 측정 지표 | 측정 방법 | 목표 |
|---|---|---|---|
| API 지연 | 엔드포인트·성공/실패별 p50, p95, p99, 최대값 | 워밍업 후 고정된 단계 부하를 주고 클라이언트 관측 시간과 서버 시간을 함께 수집 | 미정 |
| 처리량·안정성 | 초당 요청 수, 성공 수, 오류율, timeout 수 | 부하를 단계적으로 높여 포화 지점과 오류가 늘기 시작하는 지점을 확인 | 미정 |
| 포인트 락 | 락 대기 시간, 트랜잭션 시간, deadlock·lock timeout 수 | 같은 사용자 집중 부하와 서로 다른 사용자 분산 부하를 분리해 비교 | 미정 |
| DB 사용량 | 활성·대기 커넥션, 쿼리 시간, rows examined, CPU·I/O | 애플리케이션 메트릭, MySQL 성능 지표, slow query 자료를 같은 시간축으로 수집 | 미정 |
| 인기 메뉴 쿼리 | 실행 시간, 실제 처리 행 수, 인덱스 사용, 임시 테이블·정렬 여부 | 대표 데이터 분포에서 `EXPLAIN ANALYZE`와 반복 실행 결과를 기록 | 미정 |
| Outbox 전달 | 주문 커밋→최초 요청 시작과 `PUBLISHED` p50/p95/p99, backlog 건수·최고 대기 시간, 재시도율·`FAILED` 수 | 정상·지연·timeout·HTTP 오류를 주입한 Mock 외부 API로 측정 | 정상·backlog 없음 조건 최초 요청 2초 이내; 나머지 미정 |
| 애플리케이션 자원 | CPU, heap, GC, 스레드 풀·HTTP 커넥션 사용량 | API와 워커가 함께 실행되는 조건에서 부하 단계별로 수집 | 미정 |

측정 보고서에는 Java·Spring·MySQL 버전, 인스턴스 자원, DB 설정, 커넥션 풀, 데이터 건수·분포, 워커 수, 외부 API 지연, 부하 모델과 측정 시간을 함께 남겨야 합니다. 이 조건 없이 나온 수치는 서로 비교하지 않습니다.

### NFR-03 확장성과 다중 인스턴스

- 애플리케이션은 서버 로컬 세션과 로컬 락에 의존하지 않습니다.
- 로드 밸런서 뒤에서 여러 인스턴스가 같은 MySQL을 사용해 요청을 처리할 수 있어야 합니다.
- Outbox 워커는 `SKIP LOCKED`와 `claim_token` fencing으로 여러 인스턴스가 안전하게 작업을 나눕니다.
- 수평 확장이 DB 락·커넥션·외부 API 용량에 미치는 영향은 NFR-02 지표로 확인합니다.

### NFR-04 보안

- 비밀번호는 BCrypt 해시만 저장하고 원문을 로그·응답·이벤트에 남기지 않습니다.
- JWT는 HS256으로 서명하고 `sub`, `iat`, `exp`, `iss=coffee-order-system` claim을 포함합니다.
- JWT 서명 키는 `JWT_SECRET_BASE64`를 Base64 디코딩했을 때 최소 32바이트여야 하며 DB 비밀번호와 함께 소스 코드가 아닌 환경 변수로 주입합니다.
- 인증 실패 응답으로 이메일 존재 여부를 노출하지 않습니다.
- 보호 API는 본문·쿼리의 사용자 ID가 아니라 인증 주체를 사용합니다.
- 로그에 JWT, 비밀번호, DB 자격 증명을 남기지 않습니다.

`jti`, `aud`, `kid`, 애플리케이션 내부 키 회전 기능은 현재 범위에 포함하지 않습니다. 무중단 키 회전이나 여러 검증 주체가 필요해지면 별도 보안 정책과 ADR로 결정합니다.

### NFR-05 장애 복구와 전달 신뢰성

- 외부 API 장애는 주문 커밋 가능 여부에 영향을 주지 않습니다.
- Outbox 전달 의미는 at-least-once이며 수신 시스템은 `eventId`로 중복 반영을 막아야 합니다.
- 워커 중단 이벤트는 lease 만료 후 다시 처리할 수 있어야 합니다.
- `FAILED` 이벤트는 자동 삭제하지 않고 상태와 payload, 실패 시각·HTTP 상태·오류 유형을 보존합니다.
- MySQL 백업·복구, 복제, 장애 전환은 실제 운영 배포 범위가 정해질 때 별도로 결정합니다.

### NFR-06 관찰 가능성

- 구조화 로그는 요청 ID, 주문 ID, 이벤트 ID를 연계하되 민감정보는 제외합니다.
- API별 요청 수·지연·오류, DB 커넥션·트랜잭션·락, Outbox 상태·재시도·backlog를 측정할 수 있어야 합니다.
- `FAILED` 증가, backlog 최고 대기 시간 증가, deadlock·lock timeout은 운영자가 구분해 확인할 수 있어야 합니다.
- 현재 구현은 Actuator와 Micrometer로 지표를 등록하고 상세 정보를 숨긴 health endpoint만 HTTP에 노출합니다.
- 경고 임계값과 알림 채널은 실제 운영 조건이 정해진 뒤 측정 기준과 함께 승인받습니다.

### NFR-07 시간 결정성

- 인기 메뉴 집계와 재시도 시각 계산은 주입 가능한 `Clock`을 사용해 테스트할 수 있어야 합니다.
- 하나의 인기 메뉴 요청에서는 동일한 기준 시각을 모든 조건과 응답 계산에 사용합니다.
- DB·애플리케이션·테스트에서 UTC를 사용하고 시스템 기본 시간대에 의존하지 않습니다.

## 9. 구현 후 측정해 확정할 운영 기준

제품·데이터·보안 계약과 정상 조건의 Outbox 최초 요청 2초 기준은 현재 범위에서 확정했습니다. 다음 운영 수치는 실제 구현과 기준 부하 측정 뒤 별도로 승인합니다.

- 성능·용량의 정량 목표, 기준 데이터 규모와 예상 트래픽
- JDBC 풀 크기와 워커 스레드 수
- Outbox backlog·락 대기·오류율의 알림 임계값
- 24시간·30일 보존 기간이 지난 데이터의 운영 정리 절차
