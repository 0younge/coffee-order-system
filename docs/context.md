# Coffee Order Context

커피 메뉴, 포인트, 주문, 외부 이벤트와 인기 메뉴를 같은 의미로 설명하기 위한 단일 도메인 컨텍스트입니다. 코드·API·DB·테스트에서 용어가 충돌할 때 이 문서의 정의를 우선 확인하되, 구체적인 필드와 상태 코드는 [API 명세](./api-spec.md)와 [ERD](./erd.md)를 함께 따릅니다.

## 컨텍스트 범위

이 시스템은 하나의 모듈러 모놀리스 안에서 다음 책임을 가집니다.

- 요청 본문의 기존 사용자 ID 확인
- 현재 메뉴 카탈로그 조회
- 사용자의 현재 포인트 잔액 충전과 주문 결제
- 단일 메뉴 결제 완료 주문과 주문 시점 스냅샷 보존
- 주문 이벤트의 Outbox 저장, 외부 전송, 재시도 상태 관리
- 최근 7×24시간 결제 완료 주문을 기반으로 한 인기 메뉴 조회

사용자 생성·인증, 메뉴 운영, 주문 취소·환불, 포인트 원장, 외부 데이터 수집 시스템 자체는 이 컨텍스트 밖입니다. 이 경계는 배포 단위 분리를 뜻하지 않으며 현재 애플리케이션은 하나의 배포 단위입니다.

## 도메인 언어

| 표준 용어 | 한국어 표현 | 정확한 의미 | 불변식·데이터 기준 | 사용하지 않는 표현 |
|---|---|---|---|---|
| **User** | 사용자 | 시스템에 이미 존재하며 포인트를 소유하고 커피를 주문하는 식별 대상 | `users.id`와 현재 잔액이 기준 | Customer, Member |
| **Menu** | 메뉴 | 주문할 수 있는 하나의 커피 상품과 현재 이름·포인트 가격 | 현재 표시 정보는 `menus`가 기준 | Product, Item |
| **Point** | 포인트 | 커피 결제에 사용하는 정수 가치 단위. 1원은 1P | 부동소수점이 아닌 정수로 계산·저장 | Money, Credit |
| **Point Balance** | 포인트 잔액 | 사용자가 현재 사용할 수 있는 포인트 총량 | `users.point_balance`; signed `BIGINT`의 0 이상 값 | Wallet, Account balance |
| **Point Charge** | 포인트 충전 | 요청한 기존 사용자의 잔액을 양의 정수 포인트만큼 증가시키는 행위 | 같은 사용자의 잔액 변경과 직렬화, 덧셈 overflow 방지 | Deposit, Top-up |
| **Order** | 주문 | 사용자가 메뉴 하나를 선택하고 포인트 결제를 완료한 구매 기록 | 현재 범위의 주문은 모두 `PAID` | Cart, Purchase |
| **Paid Order** | 결제 완료 주문 | 포인트 차감과 주문 기록이 하나의 트랜잭션에서 함께 확정된 주문 | `orders.status = PAID` | Completed order |
| **Order Snapshot** | 주문 스냅샷 | 주문 시점의 메뉴명과 실제 결제금액을 주문에 보존한 값 | `menu_name_snapshot`, `paid_amount`; 생성 후 불변 | Current menu data |
| **Order Event** | 주문 이벤트 | 결제 완료 사실을 외부 데이터 수집 경계로 전달하는 고유 사건 | 주문과 같은 트랜잭션의 `outbox_events`; `eventId`로 식별 | Message, Log |
| **Popular Menu** | 인기 메뉴 | 기준 시각 직전 7×24시간 동안 `PAID` 주문 수가 많은 메뉴 | 주문 수는 `orders`, 응답 이름·가격은 현재 `menus`가 기준 | Trending menu, Best seller |

### 현재 메뉴 정보와 주문 스냅샷의 차이

- 메뉴 목록과 인기 메뉴 응답의 `name`, `price`는 조회 시점의 현재 `menus` 값을 사용합니다.
- 과거 주문의 `menuName`, `paidAmount`는 주문 시점의 스냅샷을 사용합니다.
- 따라서 메뉴 이름이나 가격이 향후 변경되면 인기 메뉴에 표시되는 값과 과거 주문 스냅샷이 달라질 수 있습니다. 이는 데이터 불일치가 아니라 서로 다른 시점의 의미를 보존한 결과입니다.

## 데이터의 단일 진실 원천

| 정보 | 기준 데이터 | 사용 위치 |
|---|---|---|
| 사용자 ID와 현재 포인트 잔액 | `users` | 사용자 확인, 충전, 결제 |
| 현재 메뉴명과 현재 가격 | `menus` | 메뉴 목록, 주문 시점 가격 결정, 인기 메뉴 표시 |
| 주문 당시 메뉴명과 결제금액 | `orders.menu_name_snapshot`, `orders.paid_amount` | 주문 기록, 외부 주문 이벤트의 결제금액 |
| 인기 메뉴 주문 수 | 기간 안의 `orders` 중 `status = PAID`인 행 | 인기 메뉴 순위 |
| 충전·주문의 최초 확정 HTTP 결과 | `idempotency_records` | 같은 멱등 요청 재사용 |
| 외부 전달할 주문 사건과 전달 상태 | `outbox_events` | Outbox 워커, 재시도·운영 확인 |

파생 값이나 애플리케이션 로컬 캐시는 위 기준 데이터를 대체하지 않습니다.

## 기능 경계와 의존 방향

| 경계 | 책임 | 소유 데이터 | 허용되는 주요 의존 |
|---|---|---|---|
| Menu | 현재 메뉴 목록과 인기 메뉴 조회 | `menus`; 인기 조회 시 `orders` 읽기 | Order 데이터는 집계 목적으로만 읽기 |
| Point | 요청한 기존 사용자의 현재 잔액 충전 | `users.point_balance` | Idempotency, 사용자 행 락 |
| Order | 메뉴 하나 결제, 주문 스냅샷, Outbox 생성 | `orders`, 주문과 함께 생성되는 `outbox_events` | Menu의 현재 값 읽기, Point 잔액 변경, Idempotency |
| Idempotency | 동일 논리 요청 선점, 요청 비교, 최초 결과 저장·재사용 | `idempotency_records` | Point·Order 변경의 트랜잭션 경계 안에서 동작 |
| Outbox | 이벤트 점유, 외부 전송, 재시도·보존 | `outbox_events`의 전달 상태 | 커밋된 주문 이벤트만 처리, 외부 API 호출 |

Point와 Order가 같은 `users` 행을 갱신하므로 둘 다 동일한 비관적 락 규칙을 사용합니다. 애플리케이션 인스턴스 로컬 락이나 로컬 세션은 정합성 경계가 아닙니다.

## 트랜잭션 경계

### 포인트 충전

`userId`·헤더·JSON과 충전금액이 signed `BIGINT` 범위의 양의 정수인지 검증합니다. 사용자가 없으면 멱등 레코드를 만들기 전에 거절합니다.

1. 요청 사용자·작업 유형·`Idempotency-Key` 조합을 확인합니다.
2. 요청 내용이 최초 요청과 같은지 비교합니다.
3. 사용자 행을 비관적으로 잠그고 덧셈 overflow를 검사한 뒤 잔액을 증가시킵니다.
4. 잔액 변경과 확정 응답을 같은 DB 트랜잭션으로 커밋합니다.

### 주문 및 결제

`userId`·헤더·JSON과 `menuId` 형식을 검증하고 사용자가 없으면 멱등 레코드를 만들기 전에 거절합니다.

1. 멱등 요청을 확인하고 현재 메뉴를 조회합니다.
2. 사용자 행을 비관적으로 잠근 뒤 현재 잔액을 검증합니다.
3. 포인트를 차감하고 주문 시점 메뉴명·결제금액을 `orders`에 저장합니다.
4. 대응하는 `ORDER_PAID` Outbox 이벤트를 저장합니다.
5. 포인트·주문·Outbox·멱등 결과를 한 DB 트랜잭션으로 커밋합니다.

메뉴 부재나 잔액 부족처럼 예상 가능한 비즈니스 실패는 비즈니스 데이터를 변경하지 않고 그 실패 응답을 멱등 결과로 저장해 커밋합니다. DB 연결 실패, 트랜잭션 커밋 실패, 5초 락 timeout과 deadlock 같은 인프라 오류는 비즈니스 변경과 멱등 레코드를 모두 롤백합니다.

### 외부 이벤트 전달

외부 HTTP 호출은 주문 트랜잭션 밖에서 실행합니다. Outbox 워커는 `SKIP LOCKED`로 처리 가능한 행을 고르고 `claim_token`과 30초 lease로 소유권을 표시한 뒤, 선점 트랜잭션을 커밋하고 외부 API를 호출합니다. 워커 정상·전송 가능한 기존 backlog 없음 조건에서 주문 커밋 후 2초 이내에 최초 HTTP 요청을 시작합니다. 완료 갱신은 같은 `event_id`와 `claim_token`을 가진 워커만 할 수 있어 lease가 만료된 이전 워커의 결과 덮어쓰기를 막습니다.

## 시간과 보존 의미

| 항목 | 기준 |
|---|---|
| 저장 시간대 | 애플리케이션과 MySQL 모두 UTC |
| 외부 날짜·시간 표현 | ISO 8601 UTC 문자열 |
| 인기 메뉴 구간 | `[조회 시각 - 7일, 조회 시각)`; 하한 포함, 상한 제외 |
| 멱등 결과 | `completed_at`부터 최소 24시간 보존 후 삭제 가능 |
| Outbox lease | 선점 후 30초 |
| Outbox 최초 전송 | 워커 정상·전송 가능한 기존 backlog 없음 조건에서 주문 커밋 후 2초 이내 HTTP 요청 시작 |
| `PUBLISHED` 이벤트 | 발행 완료 후 최소 30일 보존 뒤 삭제 가능 |
| `FAILED` 이벤트 | 자동 삭제 없이 보존 |

테스트는 주입 가능한 `Clock`으로 경계 시각을 고정합니다. 인기 메뉴의 “최근 7일”은 달력 날짜 7개가 아니라 정확히 7×24시간입니다.

## 기술 언어

| 용어 | 이 시스템에서의 의미 |
|---|---|
| **Pessimistic Lock** | 충전·결제 트랜잭션이 사용자 행을 잠가 같은 사용자 잔액 변경을 순서대로 처리하는 DB 락 |
| **Idempotency-Key** | 클라이언트가 하나의 논리적 충전·주문 요청에 부여하는 고유 값 |
| **Request Hash** | 같은 멱등키로 들어온 요청 내용이 최초 요청과 같은지 비교하기 위한 정규화 요청의 해시 |
| **Idempotency Record** | 최초 요청의 해시, 결과 코드, HTTP 상태, 응답 본문과 처리 상태를 저장한 레코드 |
| **Transactional Outbox** | 주문과 외부 전달 이벤트를 같은 DB 트랜잭션에 저장하고, 별도 워커가 커밋된 이벤트를 전송하는 패턴 |
| **SKIP LOCKED** | 다른 워커가 잠근 Outbox 행을 기다리지 않고 건너뛰어 여러 워커가 서로 다른 이벤트를 선점하게 하는 DB 기능 |
| **Claim Token** | 특정 선점 시도를 식별해 현재 소유자만 Outbox 결과를 갱신하도록 하는 fencing 값 |
| **Lease** | 워커가 이벤트 처리 권한을 가진 제한 시간. 만료되면 다른 워커가 다시 선점 가능 |
| **At-least-once** | 이벤트가 한 번 이상 전달됨을 보장하지만 응답 유실 등으로 중복 전송될 수 있는 전달 의미 |
| **Backlog** | 아직 `PUBLISHED`되지 않고 전송을 기다리거나 재시도 중인 Outbox 이벤트 집합 |

`Idempotency-Key`는 UUID 문자열이며, `Request Hash`는 작업 유형과 검증된 업무 값(`CHARGE|<amount>`, `ORDER|<menuId>`)을 UTF-8로 직렬화한 SHA-256 결과입니다. 사용자 ID는 멱등 유일 범위에 포함됩니다. 상세 입력 규칙은 [API 명세](./api-spec.md#11-입력-검증-규칙)를 따릅니다.

## 시스템 불변식

1. 포인트 잔액과 금액은 정수이며 포인트 잔액은 음수가 될 수 없습니다.
2. 같은 사용자의 충전과 결제는 공용 MySQL의 사용자 행 락으로 순서를 결정합니다.
3. 충전과 주문의 사용자 ID는 기존 `users.id`여야 합니다.
4. 같은 사용자·작업 유형·멱등키에 서로 다른 요청 내용을 사용할 수 없습니다.
5. 주문과 대응하는 Outbox 이벤트는 함께 커밋되거나 함께 롤백됩니다.
6. 주문 스냅샷은 주문 생성 후 현재 메뉴 변경의 영향을 받지 않습니다.
7. 외부 전송 실패는 이미 커밋된 고객 주문을 실패로 되돌리지 않습니다.
8. 인기 메뉴 순위는 주문 수 내림차순, 동률이면 메뉴 ID 오름차순입니다.

## 범위가 바뀔 때 다시 결정할 사항

- 다품목·수량·부분 취소가 필요하면 `OrderItem`과 주문 금액 모델을 다시 결정합니다.
- 감사·환불이 필요하면 현재 잔액만 저장하는 모델 대신 불변 포인트 원장과 보상 거래를 검토합니다.
- 메뉴 관리가 필요하면 판매 상태, 삭제, 과거 주문과 인기 메뉴 표시 정책을 함께 정합니다.
- 인기 집계가 측정된 병목이 되고 집계 지연을 허용할 수 있을 때만 캐시나 Kafka 조회 모델을 검토합니다.
- 사용자 생성이나 인증·인가가 필요하면 사용자 소유권과 자격 증명 정책을 새로 결정합니다.

이 항목은 현재 설계에 몰래 포함하는 미래 기능이 아닙니다. 요구가 실제로 생기면 영향과 대안을 ADR로 기록한 뒤 기존 계약을 변경합니다.
